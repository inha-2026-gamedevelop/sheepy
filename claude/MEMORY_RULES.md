- **현재 상태**: 이 저장소 전용 메모리는 **아직 비어 있다**(2026-07-11 실측 — `MEMORY.md` 인덱스도 개별 파일도 없음). 이 세션(보스 시스템 작업, git-flow 셋업, 커밋 컨벤션 신설 등)에서 나온 여러 사용자 교정·확정 결정이 아직 메모리로 저장되지 않은 상태다 — 다음 세션 착수 전 8절 기준으로 보강이 필요하다.
- **구조 규격**: `MEMORY.md`(인덱스, 매 세션 자동 로드) + 개별 사실 파일. 파일당 frontmatter로 `name`/`description`/`metadata.type`을 갖는다(3절).

## 2. 저장 기준 — 무엇을 메모리에, 무엇을 저장소에

메모리는 **이 PC의 사용자(aurenixs) 개인 자산**이다. 팀원(명진, 진욱)과 다른 PC의 세션에는 보이지 않는다. 따라서:

| 정보 성격 | 저장 위치 | 실례(있을 경우) |
|---|---|---|
| 사용자 교정·선호 (재발 방지 규칙) | 메모리 `feedback_*` | 예: "커밋 메시지에 AI 공저자(Co-Authored-By) 표기 넣지 말 것"(이 세션에 지적됨) |
| 세션 간 이어지는 개인 작업 상태 | 메모리 `project_*` | 예: "보스 씬 배선(유닛/아이콘) 미완, HANDOVER.md 참고" |
| 도구 특이사항 (재발견 비용 큰 함정) | 메모리 `project_*` | 예: "이 세션엔 Unity MCP 미연결, 매번 사용자에게 Play 확인 요청 필요" |
| **팀이 공유해야 하는 절차·스펙** | 저장소 `claude/` | `claude/HANDOVER.md`, `claude/PLAN.md`, `claude/gamedb.md` |
| 전 세션 상시 적용 규범 | 저장소 `claude/CLAUDE.md` (승격, 9절) | 코딩 컨벤션, Critical Rules |

**핵심 패턴 — 메모리는 포인터+상태, 저장소는 본문**: 메모리 파일에는 "현재 상태 한 줄 + 상세는 어디" 포인터만 두고, 절차·표·코드 스니펫 같은 본문은 저장소 문서(`claude/HANDOVER.md` 등)에 둔다. 저장소 문서 작성 시점 기준은 [체크포인트 규칙](CHECKPOINT_RULES.md) 참조.

**저장하지 않는 것**:
- 코드를 읽으면 즉시 알 수 있는 사실(함수 시그니처, 파일 위치 나열) — Grep/Glob이 항상 최신이고 메모리는 노화된다.
- 일회성 정보(특정 세션의 디버그 값, 임시 브랜치명).
- 비밀 — `Assets/StreamingAssets/KEY.txt`의 Supabase URL/ANON_KEY 원문, 계정 정보. 경로 언급만 허용. 상세는 [민감정보 기준](DATA_RETENTION_AND_PRIVACY_RULES.md).

## 3. frontmatter `type` 분류

표준 4종 — `user`(사용자 프로필) / `feedback`(교정 규칙) / `project`(작업 상태) / `reference`(외부 자료 포인터). 시스템 프롬프트에 정의된 표준 양식:

```yaml
---
name: <파일 basename과 동일, kebab-case>
description: <한 줄 요약 — 향후 세션이 관련성을 판단하는 근거이므로 구체적으로>
metadata:
  type: user | feedback | project | reference
---
```

## 4. 갱신 vs 신규 판단

**원칙: 같은 주제의 강화·구체화·상태 변화는 기존 파일 갱신. 새 파일은 새 주제일 때만.**

- 판단 절차: 저장 전에 `MEMORY.md` 인덱스를 훑고, 겹치는 항목이 있으면 그 파일을 Read → 본문에 통합 갱신한다.
- 중복 파일이 생기면(같은 주제로 두 파일) 한쪽에 통합 후 나머지 삭제 — 인덱스가 신파일만 가리키면 구파일이 고아가 되고 내용이 분산된다.
- **갱신 시 인덱스 동시 갱신 의무**: 본문을 고치면 `MEMORY.md`의 해당 한 줄 요약도 같은 턴에 고친다. 수치·상태가 본문과 인덱스에서 어긋나면(예: 버전 표류) 후속 세션이 구버전을 사실로 오인한다.

## 5. 삭제 기준

삭제는 다음 3경우에만, 반드시 **본문 통합 → 인덱스 제거 → 파일 삭제** 순으로:

1. **중복 통합 잔여**: 두 파일이 같은 주제로 판명되면 한쪽에 통합 후 나머지 삭제.
2. **수명 종료한 project 상태**: 작업이 커밋·머지되고 저장소 문서(`claude/HANDOVER.md` 등)로 내용이 완전 이관된 경우. 단, "왜 그렇게 됐는가"의 근본원인 기록은 재발 방지 가치가 있으므로 원인 요약만 남기는 갱신이 삭제보다 우선.
3. **사실 무효화**: 코드 변경으로 메모리 내용이 틀려진 경우. 틀린 메모리는 없는 메모리보다 해롭다 — 발견 즉시 갱신 또는 삭제.

**feedback 타입은 원칙적으로 삭제하지 않는다.** 사용자 선호는 만료되지 않으며, `claude/CLAUDE.md`로 승격(9절)돼도 Why 이력의 출처로 남긴다. 사용자가 명시적으로 규칙을 철회한 경우에만 삭제한다.

## 6. [[링크]] 규칙

**표준: 파일 basename(확장자 제외, kebab-case)으로 통일한다.** 링크는 **탐색 비용을 줄일 때만** 건다 — 모든 관련 항목을 나열하는 장식성 링크는 금지. `[[name]]`이 아직 없는 메모리를 가리켜도 무방하다 — 앞으로 써야 할 메모리의 표시로 취급한다.

## 7. 상대날짜 → 절대날짜 변환

**"오늘 / 어제 / 방금 / 최근 / 지난주" 금지. 반드시 `YYYY-MM-DD`로 쓴다.**

근거: 메모리는 작성 후 수 주 뒤에 읽히며, 하네스가 "This memory is N days old" 같은 노화 경고를 붙일 정도로 시차가 기본 전제다. "어제 수정함"은 24일 뒤에 무의미하지만 "2026-07-11 수정, 미커밋"은 영원히 유효하다. 상태 서술에도 동일 적용: "미커밋"처럼 시간이 지나면 변하는 상태는 날짜를 병기한다.

## 8. 교훈(lessons) 축적 규칙 — 사용자 교정 시 Why / How to apply 필수

공식 프롬프팅 가이드는 lessons를 "한 줄 요약 + **이유**를 포함한 markdown 파일"로 저장하고 파일 기반 메모리 표면을 제공할 때 성능이 향상된다고 명시한다.

**트리거**: 사용자가 산출물·행동을 교정하거나(같은 지적 2회째가 아니라 **1회째부터**), 비일반적인 접근을 확인해주면(성공 확인도 저장 대상) 그 턴 내에 `feedback_*` 메모리를 생성 또는 갱신한다.

**필수 2필드**:
- **Why**: 교정이 발생한 **구체 사건**(날짜, 무엇이 어떻게 잘못됐는지)과 가능하면 **사용자 발언 원문 인용**. "사용자가 싫어함" 수준의 요약은 후속 세션이 규칙의 적용 범위를 판단할 수 없게 만든다.
- **How to apply**: 다음 작업에서 행동을 어떻게 바꾸는지를 **콜사이트 수준**으로. "주의한다"가 아니라 "무엇을 대신 쓴다 / 어느 시점에 무엇을 확인한다".

**이 세션에서 실제로 발생해 메모리화가 필요한 사례 (2026-07-11, 미저장 — 다음 세션 착수 시 우선 기록)**:
1. 커밋 메시지에 `Co-Authored-By: Claude ...` 줄을 넣지 말 것 — 사용자가 커밋 직후 명시적으로 지적, 직전 커밋을 amend로 정정한 전례.
2. "커밋만 하고 푸시하지 마" — push는 사용자가 별도로 명시할 때만.
3. 커밋 컨벤션은 `type(scope): 제목`이 아니라 `type: Scope 제목`(괄호 없이) 형식을 선호.
4. HANDOVER.md는 팀 문서 전체 이력을 담기보다 "이번에 고친 것 / 해야 할 것" 위주로 간결하게 유지하라는 지적.

## 9. `claude/CLAUDE.md` 승격 파이프라인

feedback 메모리가 **전 세션·전 작업에 상시 적용되는 규범**으로 굳으면 저장소 `claude/CLAUDE.md`(및 [프로젝트 지침 통합 재작성](PROJECT_INSTRUCTIONS_V2.md))로 승격한다.

- 승격 기준: 조건 없이 항상 적용 + 팀원에게도 유효(개인 선호가 아님) + 한 줄로 요약 가능.
- 승격 후: 메모리 파일은 Why 출처로 **유지**하되, `claude/CLAUDE.md`가 규범의 단일 출처다. 200줄 내외 권장을 고려해, 승격은 한 줄 규칙만 올리고 사건 서사는 메모리에 남긴다.
- `claude/CLAUDE.md` 원본은 직접 수정하지 않고, 재작성 제안은 [SYSTEM_PROMPT_V2.md](SYSTEM_PROMPT_V2.md) 경유한다는 것은 **이 프로젝트에서는 완화된 규칙**이다 — 실제로 이번 세션에서는 사용자 승인하에 CLAUDE.md를 직접 수정한 전례가 있다(커밋 컨벤션 준수 규칙 추가 등). 원본 직접 수정 여부는 사용자 선호에 따른다.
