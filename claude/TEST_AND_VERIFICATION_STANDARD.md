# 테스트·검증 표준

> 목적: 이 프로젝트에서 "검증했다"고 말하기 위해 실제로 실행해야 하는 표준 절차를 정의한다. | 공식 근거: Anthropic 프롬프팅 가이드(진행 보고는 도구 결과 근거 의무) + 프로젝트 내부 규칙 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 재작성(ENTRY_SCENE 치트 플로우 전제를 MCP 연결/미연결 듀얼 경로로 교체)

---

## 1. 대원칙 — 검증 경로는 Unity MCP 연결 여부로 갈린다

이 프로젝트에는 고정된 "치트 진입 플로우"나 자동 테스트 봇이 없다(2026-07-11 기준 미확인). 대신 검증 경로는 **Unity MCP가 이번 세션에 연결돼 있는가**로 갈린다.

- **연결 시**: 에디터를 직접 조작해 컴파일/Play/콘솔 로그를 능동적으로 확인할 수 있다.
- **미연결 시(이번 세션 기본 상태)**: 코드 리뷰까지만 스스로 하고, 실제 컴파일·런타임 동작 확인은 **사용자에게 구체적으로 요청**한다. 이는 결함이 아니라 이 프로젝트의 정상적인 검증 경로다 — "미검증"을 숨기지 않고 명시하는 것이 규칙 준수다.

역방향 함정 금지: 코드가 논리적으로 맞아 보인다고 "검증됨"으로 보고하지 않는다. 검증으로 인정되는 것은 실제 콘솔 로그나 사용자의 명시적 확인뿐이다([진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md) 1절과 동일 원칙).

## 2. 검증 수준 3단계

[진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md)의 증거 등급 1·2·3과 1:1 대응한다.

### L1 — 컴파일 검증 (모든 .cs/씬/프리팹 수정의 최소선)

- **MCP 연결 시**: 수정 후 에셋 리프레시 → 컴파일 완료 대기 → `read_console`(Error 필터)로 **에러 0건**을 도구 출력으로 확인.
- **MCP 미연결 시**: 각 파일을 다시 훑어 문법·타입·네임스페이스·using 누락을 확인하고, "실제 컴파일은 미확인 — 에디터 확인 필요"를 보고에 명시.

### L2 — 기능 검증 (로직 변경 시)

목표 씬(예: `Boss.unity`)에서 수정한 기능이 **1회 이상 동작**했음을 관찰한다.

- MCP 연결 시: `read_console` 로그, 필요 시 스크린샷 도구.
- MCP 미연결 시: 사용자에게 "어느 씬에서, 무엇을 눌러서, 무엇을 확인해야 하는지"를 구체적으로 요청하고, 사용자의 답변을 근거로 삼는다. 실제로 이번 세션에서 감정 자동 전환은 이 방식으로 사용자가 "로그가 떠" 확인을 답변한 것을 근거로 검증됐다.
- 로그·화면으로 판별 불가능한 시각 요소(정확한 색상, 미세 배치)는 "미검증 — 사용자 화면 확인 필요"로 보고한다.

### L3 — 리와인드 왕복 검증 (리와인드 참여 구조 변경 시)

리와인드 참여 오브젝트(`IRewindable` 구현, 결정 로그, 스냅샷)를 새로 만들거나 바꿨다면, **되감기 → 재생 왕복** 후 같은 결과가 재현되는지 확인한다. 세이브 저장/재로드 왕복에 대응하는 이 프로젝트의 검증이다.

### 작업 유형별 요구 수준 (요약)

| 작업 유형 | 요구 수준 |
|---|---|
| 문서·주석만 | 수준 외 (단, 인용 경로 전부 실존 확인) |
| 로그 문구·GameDB 기본값 | L1 |
| `Minsung.*` 로직 수정 | L2 이상 |
| 씬/프리팹 YAML 편집 | L2 + 에디터 재확인 요청 |
| 매니저 초기화·씬 전환 경로(예: `BossHandoff`) | L2~L3 |
| 리와인드 스냅샷·결정 로그 | L3 (되감기 왕복 필수) |

전체 표와 미달 시 보고 문구("등급 2까지 검증됨, 플로우 검증 미실시")는 [진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md) 3절이 원본이다. 검증을 별도 컨텍스트에 위임하려면 [검수 Subagent 사양서](VERIFIER_SUBAGENT_SPEC.md)의 검수자를 쓴다(사용자 요청 시).

## 3. Unity MCP 연결 확인 절차

매 세션 시작 시(또는 검증이 필요한 시점에) 다음을 확인한다:

1. `ToolSearch`로 "unity" 관련 도구를 검색해 연결 여부 확인.
2. 연결 정보는 `claude/PLAN.md`에 근거가 있다 — `http://127.0.0.1:8080/mcp`(Unity 패키지 내장 서버). Unity 에디터가 켜져 있어야 하고, MCP 설정이 stdio가 아니라 이 주소를 가리켜야 한다.
3. 미연결이면 재시도로 시간을 낭비하지 말고([FALLBACK_POLICY.md](FALLBACK_POLICY.md)) 바로 코드 리뷰 + 사용자 확인 요청 경로로 전환한다.

## 4. 사용자에게 검증을 요청하는 방식 (MCP 미연결 시 표준)

MCP가 없을 때 검증 요청은 다음을 포함해야 실효성이 있다:

1. **어느 씬**에서 확인해야 하는지(예: `Boss.unity`).
2. **무엇을 눌러야** 재현되는지(예: "1페이즈 근접전 중 Shift 홀드").
3. **무엇을 관찰해야** 하는지(예: "Console에 `[BossController] 감정 전환 -> ...` 로그가 뜨는지").
4. 실제로 이번 세션에서 이 방식으로 슬로우모션 버그, 감정 전환, 씬 이관을 순차 검증했다 — 매번 이 4요소를 갖춘 질문이 빠른 회신을 받았다.

## 5. 검증 결과 보고

검증을 어디까지 했는지는 상태 어휘("완료(검증됨)"/"구현됨(미검증)" 등 — [진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md) 6절)로 서술하고, 수정 위치는 전부 `[파일:라인](경로#L라인)` 링크로 제시한다. **도구 출력이나 사용자 확인이 없는 검증 주장은 보고에 쓸 수 없다.**
