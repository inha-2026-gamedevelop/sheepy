# 대화 분리 기준

> 목적: 긴 세션을 언제 끊고 새 세션으로 넘어갈지, 그리고 넘어갈 때 무엇을 남겨야 다음 세션이 손실 없이 재개하는지를 성문화한다. | 공식 근거: Anthropic 프롬프팅 가이드(파일 기반 메모리·fresh-context 검증자 우위) + Claude 사용량/컨텍스트 공식 문서(support.claude.com) + 프로젝트 내부 규칙 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

## 1. 전제 — 세션이 길어지면 무슨 일이 생기는가

### 1-1. 공식 수치 (인용 가능한 것만)

컨텍스트 윈도·사용량 한도 등 모델별 수치는 시점에 따라 바뀌므로 이 문서에 하드코딩하지 않는다. 필요 시 [CONTEXT_LIMITS_POLICY.md](CONTEXT_LIMITS_POLICY.md)와 공식 문서(support.claude.com, code.claude.com)에서 조회한다. 확실한 공식 사실만 아래에 남긴다.

| 항목 | 사실 |
|---|---|
| Claude Code 압축(compact) | 대화가 컨텍스트 한계에 접근하면 이전 메시지를 자동 요약해 세션을 이어간다 |
| 압축 트리거 임계값 | **공식 수치 미확인** |
| 대화당 최대 메시지 수 | **공식 수치 미확인** |

### 1-2. 왜 "요약으로 버티기"보다 "분리"가 나은가

압축(요약)은 대화를 이어주지만 **요약에서 탈락한 세부가 곧 이 프로젝트의 리스크**다. 이 저장소의 작업 단위는 "어떤 파일이 미커밋 상태인가", "사용자가 어떤 선택지를 승인했는가" 같은 요약에 잘 살아남지 않는 종류의 사실에 의존한다(§2-3 참조).

공식 가이드도 같은 방향을 가리킨다: 장시간 작업에서는 **오염되지 않은 새 컨텍스트(fresh-context)의 검증자가 자기비판보다 우수**하며, **파일 기반 메모리 표면을 제공하면 성능이 향상**된다. 즉, 이 프로젝트의 올바른 지속 전략은 "한 세션을 끝까지 늘리기"가 아니라 **"상태를 파일(메모리·HANDOVER·커밋)로 내려놓고 새 세션으로 갈아타기"**다. 새 세션은 `claude/CLAUDE.md`와 메모리 인덱스(`MEMORY.md`)를 자동으로 로드하므로, 파일에 내려놓은 것만이 다음 세션의 출발 자산이 된다.

## 2. 분리 트리거 3종

### S1. 작업 단위 완결 시 (권장 분리점)

작업 하나가 "검증 → HANDOVER/메모리 갱신 → 커밋(또는 미커밋 사유 기록)"까지 닫혔으면 그 지점이 가장 싼 분리점이다. 이 프로젝트의 실제 완결 단위 예:

- 보스 기능 1건(예: 감정 자동 전환 구동부) 구현 + 커밋 완료
- `claude/HANDOVER.md`의 "이번에 고친 것" 목록이 검증까지 마쳐진 상태

완결 직후에도 다음 작업을 같은 세션에서 이어가고 싶어지지만, 다음 작업이 **다른 파일군**을 건드린다면 S2와 동일하게 취급하고 분리한다.

### S2. 주제 전환 시

앞 주제의 파일 수십 개가 컨텍스트에 로드된 상태로 무관한 주제를 시작하면, (a) 앞 주제의 파일 상태가 새 주제의 판단을 오염시키고 (b) 컨텍스트만 낭비된다. 이 프로젝트에서 자주 겪는 전환 축:

| 전환 예 | 판단 |
|---|---|
| 보스 시스템(`03.Boss/`) 작업 중 → AI_Workflow_TemplatePack 문서 이식 요청 | **분리 권장** — 파일군·검증 방식이 전혀 다름 (실제로 이 세션에서도 이 전환이 있었다) |
| 1페이즈 기믹 개선 → 2페이즈 씬 이관 | 같은 주제(Boss), 유지 가능 (단 S1 완결점이면 분리 우선) |
| 코드 작업 중 → 팀원(명진/진욱) 소유 코드 이슈 발견 | 분리 이전 문제 — 소유자 확인 없이 진행 금지이므로 [작업 경계선](ACTION_BOUNDARIES.md)에 따라 HANDOVER로 인계 |

### S3. 압축 후 품질 저하 신호 감지 시 (즉시 분리)

컨텍스트 요약(압축)이 발생한 뒤 아래 신호가 하나라도 보이면, 그 세션의 판단은 더 이상 신뢰하지 말고 §3 체크리스트를 실행한 뒤 분리한다.

**신호 A — 이전 결정 재질문/역행.** 사용자가 이미 확정한 결정을 다시 묻거나 뒤집는다. 실제 결정 사례: "브랜치 전략은 git-flow `feature/<기능명>` 방식으로", "커밋 메시지에 AI 공저자 표기 넣지 말 것"(이 세션에서 명시적으로 지적됨). 압축 후 이런 확정 사항을 재론하기 시작하면 요약에서 결정이 탈락한 것이다.

**신호 B — 파일 상태 착각.** 미커밋 변경을 커밋된 것으로 취급하거나, 이미 폐기된 구조를 현행으로 언급한다. 이 저장소의 대표 함정:

- 브랜치를 착각: `main`/`develop`/`feature/*` 중 어디서 작업 중인지 재확인 없이 단정.
- 이미 수정한 파일(예: `HitStopController.cs`의 코루틴 재시작 로직)이 여전히 옛 버전이라고 전제하고 재분석 시작.

**신호 C — 규칙·검증 절차 이탈.** 중괄호 생략, 후위 연산자, `Resources.Load` 직접 호출을 금지 사항으로 잘못 판단하는 등(이 프로젝트는 GameDB가 `Resources.Load` 자동 로드를 공식 패턴으로 씀 — 다른 프로젝트 관례를 혼동하지 말 것) `claude/coding-convention.md` 규약을 세션 초반에는 지키다가 어기기 시작하는 경우. 규칙이 요약에서 희석된 신호다.

신호 감지 시 해당 세션에서 새로 내린 결론은 [진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md) 기준의 증거(도구 출력)가 있는 것만 인수하고, 증거 없는 주장은 이관 문서에 "미검증"으로 적는다.

## 3. 세션 이관 체크리스트 (분리 확정 시 이 순서대로)

**① 메모리 갱신.** 이번 세션에서 바뀐 상태를 영속 메모리(`C:\Users\inha\.claude\projects\d--unity2d\memory\`)에 반영하고, `MEMORY.md` 인덱스도 함께 고친다. 새 세션이 자동으로 읽는 것은 이 인덱스뿐이므로, 여기 없는 사실은 없는 사실이다. 작성 형식은 [메모리·교훈 규칙](MEMORY_RULES.md).

**② 필요시 HANDOVER 작성/갱신.** [체크포인트 규칙](CHECKPOINT_RULES.md)의 트리거에 해당하면 `claude/HANDOVER.md`를 표준 목차로 갱신한다. 한 세션에 완결된 소규모 작업은 HANDOVER 없이 ①의 메모리 한 줄로 충분하다.

**③ 미커밋 변경 상태 보고.** `git status`를 실제로 실행해 그 출력을 근거로 보고한다(추측 금지). 보고에 포함할 것:

- 이 세션 작업분 파일 목록과 각각의 의도 (커밋 전 검증이 남았으면 그 사실)
- **이 세션 작업분이 아닌** 변경 — 병행 작업자(팀원, 또는 사용자가 IDE에서 직접 한 작업)의 변경은 구분해 명시하고 커밋 대상에서 제외
- 커밋할 것이면 `claude/commit-convention.md` 형식(`type: Scope 한국어 제목`)으로. push는 사용자가 명시적으로 요청할 때만 (Git Safety Protocol).

**④ 새 세션 첫 프롬프트 요약 작성.** 아래 §4 템플릿으로, 사용자가 새 세션에 붙여 넣을 텍스트를 이관 세션이 직접 산출한다. `claude/CLAUDE.md`·메모리 인덱스는 자동 로드되므로 거기 있는 내용은 반복하지 않고 **포인터 + 이번 작업 고유 상태**만 담는다.

## 4. 새 세션 첫 프롬프트 템플릿

공식 권고 "요청의 의도/배경을 함께 제공" 패턴을 따른다. 골격:

```text
[배경] <큰 목표 1문장 — 무엇을 위해 이 작업을 하는가>
[읽을 문서] <재개 기준 문서 경로 — claude/HANDOVER.md 우선, 없으면 claude/PLAN.md>
[현재 상태] <마지막 검증 지점 1~2문장 + 미커밋 파일 목록(git status 근거)>
[확정된 결정] <재론 금지할 사용자 결정 1~3개>
[다음 작업] <구체 요청 1개>
[주의] <이 작업의 경계·함정 — 없으면 생략>
```

작성 예 (보스 시스템 작업 이관, 이번 세션 실상태 기준):

```text
[배경] 보스(Azathoth) 기획서 대비 구현 gap을 메우는 중이다.
[읽을 문서] claude/HANDOVER.md 부터. 상세 이력은 claude/PLAN.md Boss 섹션(HIGH/MED/LOW).
[현재 상태] 감정 자동 전환·씬 이관 로직 구현 완료(feature/boss-emotion-scene-handoff 브랜치 커밋됨).
  보스 유닛(_phase1Clones/_body/_heartPickup) 씬 배선은 미완 — 배치돼 있지 않아 근접 패턴을 눈으로 확인 불가.
[확정된 결정] 3페이즈 전용 맵 씬은 명진이 새로 제작(기존 Boss.unity 대체). 보스 애니메이터 연동은 진욱 담당.
[다음 작업] BossEmotionHUD._icons에 아이콘 등록 후 Play모드 재검증.
[주의] Unity MCP 미연결 — 코드 수정 후 컴파일/Play 확인은 사용자에게 요청할 것.
```

의뢰 양식의 일반형은 [프롬프트 표준 템플릿](PROMPT_TEMPLATE_STANDARD.md) · [작업 목적 설명 템플릿](REQUEST_CONTEXT_TEMPLATE.md)을 따른다.
