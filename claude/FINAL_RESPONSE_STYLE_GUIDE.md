# 최종 답변 스타일 가이드

> 목적: 이 저장소에서 AI가 사용자에게 내놓는 최종 답변의 형식을 성문화하여, 모델이 바뀌어도 동일한 보고 품질을 유지한다. | 공식 근거: Anthropic 프롬프팅 가이드(사용자 커뮤니케이션 스타일·진행 보고 근거화) + 프로젝트 내부 규칙(CLAUDE.md, 영속 메모리) | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

---

## 1. 왜 이 문서인가

공식 프롬프팅 가이드는 장시간 작업 후 최종 요약을 **"새로운 reader" 기준으로 재구성**하고, **outcome first + 완전 문장 + 스펠아웃**으로 쓰라고 권고한다. 도구 호출 사이에 쓰는 "working shorthand"와 최종 답변은 다른 문체라는 것이 공식 입장이다. 또한 진행 보고는 **툴 결과로 각 주장을 감사(audit)** 한 뒤에만 하라고 권고한다.

완료 주장 자체의 증거 요건은 [진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md)이, 답변에 담을 수 있는 행동의 한계는 [작업 경계선](ACTION_BOUNDARIES.md)이 규정한다 — 이 문서는 "어떤 형식으로 말하는가"에 한정한다.

## 2. 스타일 규칙 8조

### 제1조 — 한국어, 시니어 Unity 개발자 톤
`claude/CLAUDE.md` `Communication Protocol` 절이 최상위 규약이다: 모든 대화 응답·설명·질문은 **한국어**, 톤은 전문적·간결·기술적으로 정확하게. 코드 식별자·경로·로그는 원문 그대로 인용한다.

### 제2조 — 결론 우선 (outcome first)
최종 답변의 첫 1~2문장에 **결과 또는 판정**을 놓는다. "무엇을 했고, 지금 상태가 무엇인지"가 먼저, 과정 설명은 그 뒤. 나쁜 패턴은 조사 과정을 시간순으로 서술하다가 마지막 문단에 결론을 두는 것 — 사용자는 첫 화면에서 판정을 읽을 수 있어야 한다.

### 제3조 — 수정 코드는 전수 하이퍼링크
코드를 수정했으면 **요청받지 않아도, 모든 수정 위치를 빠짐없이** 다음 형식으로 제시한다.

```
[HitStopController.cs:94-116](Assets/01.Scripts/04.TimeSystem/HitStopController.cs#L94-L116)
```

- 라인 번호는 **편집 후 최종 위치** 기준. 대량 편집 뒤에는 Grep으로 최종 라인을 재확인하고 링크한다(추정 금지).
- 수정한 메서드/블록 단위로 전부 링크한다. 대표 한두 개만 링크하고 끝내지 않는다.
- 답변 작성 직전 자가 점검: "이 턴에 Edit/Write 한 위치 중 링크 안 단 곳이 있는가?"

### 제4조 — 표는 짧은 열거에만
표 셀에는 식별자·경로·상태·한 줄 값만 넣는다. 인과 설명, 조건 분기, 여러 문장이 필요한 내용은 본문 문단이나 목록으로 쓴다. 좋은 용례: 수정 파일 목록, 완료/잔여 현황. 나쁜 용례: "원인 분석" 열에 세 문장짜리 설명을 밀어 넣은 표.

### 제5조 — 화살표 체인으로 서술 대체 금지
`정지 → 재시작 → 복원 스킵` 같은 화살표 체인은 세션 중의 working shorthand이지 최종 답변 문체가 아니다. 최종 답변에서는 완전한 문장으로 풀어 쓴다. **예외**: 프로젝트 공인 표기 인용 — `claude/CLAUDE.md`의 아키텍처 다이어그램(`PlayerInput(입력) -> PlayerMovement / ...`)은 그 자체가 공식 표기이므로 그대로 인용한다.

### 제6조 — 임의 약어 금지
`claude/CLAUDE.md`와 프로젝트 문서에 이미 정착된 약어(GameDB, SO, HUD, BT 등)만 사용한다. 세션 중 즉석에서 만든 축약(예: `BossController`를 "보컨"으로)은 최종 답변에 쓰지 않는다. 클래스·파일은 항상 실명으로 부른다 — 다음 세션이 Grep으로 바로 찾을 수 있는 이름이어야 한다.

### 제7조 — 긴 세션 후 요약은 재접지(re-grounding) 방식
수십 턴이 지난 세션의 최종 요약은 직전 턴의 연속으로 쓰지 않는다. **이 대화를 처음 보는 reader**(다음 세션, 또는 며칠 뒤의 사용자)를 기준으로 다음 순서로 재구성한다.

1. 목표 재진술 — 이 작업이 원래 무엇을 위한 것이었는지 1~2문장.
2. 결과 — 무엇이 어떤 상태로 끝났는지 (제2조와 동일하게 앞에).
3. 검증 — 어떤 증거(코드 리뷰, 사용자 확인, Play 로그)로 확인했는지.
4. 잔여·후속 — 남은 일과, HANDOVER를 갱신했다면 그 사실.

세션 중반에만 등장했던 용어·변수명을 정의 없이 요약에 쓰면 재접지 실패다. `claude/HANDOVER.md` 작성 시점·목차는 [체크포인트 규칙](CHECKPOINT_RULES.md)을 따른다.

### 제8조 — 미검증 주장은 미검증이라고 쓴다
"완료"는 도구 결과나 사용자 확인으로 확인된 것에만 붙인다. 확인 못 한 항목은 hedging으로 얼버무리지 말고 **"미검증"으로 명시**한다. 예: "코드 리뷰상 문법 오류 없음, 런타임 동작은 미검증(에디터 Play 확인 필요)". 세부 증거 기준은 [진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md) 참조.

## 4. 기존 세션 관례와 연계된 보고 규칙

### 관례 1 — Play 검증 요청 보고
Unity MCP가 연결되어 있지 않은 것이 이 세션들의 기본 상태다. 코드 수정 후에는 다음을 명시한다.

1. 코드 리뷰 관점에서 확인한 것(문법·타입·네임스페이스)과 확인하지 못한 것(실제 컴파일·런타임 동작)을 구분.
2. 무엇을, 어떤 씬에서, 어떤 순서로 확인해야 하는지 구체적으로 요청.
3. Unity MCP가 연결되면 이 규칙이 [테스트·검증 표준](TEST_AND_VERIFICATION_STANDARD.md)의 MCP 경로로 전환된다.

### 관례 2 — 기획 판단이 걸린 답변은 질문으로 끝난다
스펙 갭이나 A/B 선택이 필요한 발견은 **수정 착수 보고가 아니라 "영향 요약 + 질문"으로 끝나야 한다**(3-2 예시의 끝맺음이 표준형). 사용자가 승인하면 결정 사항을 `claude/HANDOVER.md`나 `claude/PLAN.md`에 기록하고, 이후의 수정 보고에는 그 결정을 반영했다는 사실을 포함한다.

### 관례 3 — 커밋 보고에는 제외 목록
커밋 실행 보고에는 커밋한 파일 목록과 함께, 이 세션이 만들지 않은 변경(사용자가 IDE에서 병행 작업한 파일 등)을 발견했다면 **스테이징에서 제외한 목록**을 명시한다.

## 5. 금지 패턴 대조표

| 금지 패턴 | 교정 |
|---|---|
| "수정했습니다. 확인해보세요." (링크 없음) | 제3조 형식으로 전 수정 위치 링크 |
| "정지→재시작→복원 스킵" 식 원인 설명 | 완전 문장으로 인과 서술 (제5조) |
| "아마 잘 될 겁니다" / "~일 수도 있습니다" 반복 | 검증됨/미검증을 이분해 명시 (제8조) |
| 요약에 세션 중 임시 약어("그 매니저") | 실명 클래스/경로로 치환 (제6조) |
| 결론이 마지막 문단에 등장 | 첫 1~2문장으로 이동 (제2조) |
| 분석 요청에 무단 수정 후 "수정까지 해뒀습니다" | 판정만 보고하고 정지, 수정은 승인 후 ([작업 경계선](ACTION_BOUNDARIES.md)) |
