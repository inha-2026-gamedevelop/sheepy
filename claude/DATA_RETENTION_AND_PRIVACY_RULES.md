# 민감정보 기준

> 목적: AI 세션이 외부 채널(Artifact·웹 검색·외부 API)로 내보낼 수 있는 정보의 등급과 판단·대응 절차를 정의한다. | 공식 근거: Anthropic 데이터 보존 정책(모델별 상이, 공식 문서 확인 요망) + 프로젝트 내부 규칙 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

---

## 1. 전제: "대화에 올리는 것"과 "외부에 게시하는 것"은 다르다

모델별 데이터 보존 정책(보존 기간, zero data retention 지원 여부)은 시점·모델에 따라 다르므로 이 문서에 하드코딩하지 않는다. 필요 시 공식 문서(Anthropic 개인정보/보존 정책 페이지)에서 확인한다. 실무적으로는 두 층위로 나눠 다룬다.

| 층위 | 의미 | 기본 방침 |
|---|---|---|
| 컨텍스트 주입 | Read/Grep 결과가 대화에 포함됨 | 작업에 필요한 만큼만. 절대 금지 등급은 **값 대신 경로·구조만** 컨텍스트에 올림 |
| 외부 게시 | Artifact 게시, 웹 검색 쿼리, 외부 API(MCP 커넥터) 전송 | 아래 등급표에 따라 통제. 절대 금지 등급은 무조건 차단 |

이 세션 환경에 실제 연결된(또는 연결 가능한) 외부 채널:

- **Artifact 도구** — claude.ai 호스팅 웹 페이지. 기본 비공개지만 사용자가 링크 공유 가능 → "게시"로 취급.
- **WebSearch / WebFetch** — 검색 쿼리·fetch URL 자체가 외부 서버로 전송됨.
- **Gmail / Google Drive / Google Calendar MCP 커넥터** — 이 프로젝트에는 미인증 상태(연결 시 명백한 외부 전송으로 취급).
- **Unity MCP** — 로컬 Unity Editor 연동이므로 외부 채널이 **아님**(자유 사용, 연결된 경우).
- git push — GitHub 저장소(`inha-2026-gamedevelop/unity2d`)가 private인지 확인 필요. private이라도 팀 외 유출 방지를 위해 커밋 메시지에 절대 금지 값(2절) 인용은 금지(원격 이력은 삭제 곤란).

## 2. 절대 금지 목록 (어떤 외부 채널로도 전송 금지)

### 2-1. Supabase 크리덴셜

- 위치: `Assets/StreamingAssets/KEY.txt`(`.gitignore`에 등록됨) — `URL=`/`ANON_KEY=` 형식으로 Supabase 프로젝트 URL과 anon key를 담는다. 랭킹/고스트 리플레이 백엔드(`SupabaseClient.cs`)가 런타임에 로드한다.
- **키 값은 이 문서를 포함한 어떤 산출 문서·Artifact·커밋 메시지·검색 쿼리·외부 API 페이로드에도 재현 금지.** "KEY.txt의 ANON_KEY"처럼 경로·필드명으로만 지칭한다.
- `.gitignore`로 이미 커밋 대상에서 제외돼 있지만, 실수로 `git add -f` 등으로 포함되지 않았는지 커밋 전 확인한다.

### 2-2. 계정·이메일

- 담당자 이메일 주소, git 사용자 계정 정보, Anthropic/Google 계정 관련 값 일체. 이 운영 문서 30종에도 실값을 쓰지 않는다(역할명 "사용자", "담당 팀원"까지만 허용).
- `.claude/settings.json` / `.claude/settings.local.json`은 로컬 절대 경로·권한 목록을 포함하므로 원문 전체를 Artifact 등에 게시하지 않는다(구조 설명은 가능).

### 2-3. 미공개 게임 콘텐츠/아트 에셋

- 스토리·대사·기획 원본이 저장소 밖 문서(예: 사용자가 대화로만 전달한 기획서)로 존재한다면, 갭 분석 요약은 저장소 문서(`claude/PLAN.md` 등)에 남기되 원본 전체 전재는 지양한다.
- 아트·사운드 에셋(`Assets/09.Visual/`, `Assets/05.Sounds/` 등 미공개 게임 리소스)의 이미지·오디오 파일을 Artifact에 data URI로 임베드하거나 외부로 전송하지 않는다. 스크린샷 기반 UI 디버깅이 필요하면 사용자에게 채널을 먼저 확인한다.

## 3. 등급 분류

| 등급 | 대상 (이 저장소의 실물) | 허용 범위 |
|---|---|---|
| **공개 가능** | 코드 구조·컨벤션: `claude/coding-convention.md`의 네이밍 규칙, `PersistentSingleton<T>` 같은 일반 패턴, "Unity 2D 코루틴 리스타트 버그" 류의 **일반화된** 기술 질문 | 웹 검색 쿼리·Artifact 게시 가능. 단 프로젝트 고유명사 최소화(4절 ③) |
| **내부 한정** | 기획·설계: `claude/README.md`(보스 기획 상세), `claude/PLAN.md`, `claude/HANDOVER.md`, GameDB(SO) 밸런싱 값, 이 AI_Workflow 문서군 | git·팀 내부 공유만. Artifact 공유 링크 생성 지양, 외부 API 전송 금지, claude.ai Projects 지식 업로드는 사용자 승인 후 |
| **절대 금지** | 2절의 3종: Supabase 크리덴셜, 계정·이메일, 미공개 콘텐츠/아트 원본 | 모든 외부 채널 차단 + 컨텍스트 주입도 최소화(값 대신 경로) |

판단이 애매한 자료(예: 신규 문서, 외부 협력사 자료)는 **한 등급 위로** 취급하고 사용자에게 질문한다.

## 4. 외부 서비스 사용 전 판단 절차 (5단계)

전송/게시 도구 호출 직전에 순서대로 점검한다.

1. **금지값 스캔**: 페이로드에 2절 3종(크리덴셜·계정·미공개 원본)이 있는가 → 있으면 즉시 제거하거나 호출 포기.
2. **수신자 확인**: 내부 한정 등급이 포함되면 수신자가 팀 내부인가(Gmail 초안의 수신 주소, Drive 폴더 위치 등). 외부이거나 불명이면 전송하지 않는다.
3. **쿼리 일반화**: WebSearch/WebFetch 쿼리에서 프로젝트 고유명사(The Last Re:wind, 클래스명 등)와 수치를 제거하고 기술 일반형으로 바꾼다.
4. **Artifact 게시 전 재검토**: 공개 가능 등급만 담겼는지 확인. Artifact는 게시 후 사용자가 공유할 수 있으므로 "어차피 비공개"라는 가정을 하지 않는다.
5. **불확실하면 질문**: 판단이 서지 않으면 전송하지 않고 사용자에게 묻는다. [작업 경계선](ACTION_BOUNDARIES.md)의 "실행 전 질문" 원칙과 동일 계열이다.

## 5. 위반 의심 시 대응

1. **즉시 보고 (작업 중단보다 우선)**: 진행 중 작업을 멈추고, *무엇이 · 어떤 채널로 · 언제* 나갔는지를 도구 호출 기록 기준으로 사용자에게 보고한다. 추정과 확인 사실을 구분해 서술한다 — [진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md)의 증거 기반 보고 원칙을 그대로 적용.
2. **게시물 삭제 요청**: Artifact에 유출된 경우 해당 내용을 제거한 정화 버전으로 같은 URL에 즉시 재배포하고, 사용자에게 기존 공유 링크 회수·게시물 삭제를 요청한다.
3. **크리덴셜 노출 의심 시 특칙**: Supabase 키가 노출됐다면 즉시 재발급이 필요할 수 있음을 사용자에게 알린다. 에이전트가 임의로 키를 교체하거나 계정을 조작하지 않는다.
4. **사례 기록**: 처리 완료 후 [메모리·교훈 규칙](MEMORY_RULES.md)에 따라 사고 원인과 재발 방지 규칙을 메모리에 1건 남긴다.

## 6. 공식 근거의 한계

이 문서가 인용하는 데이터 보존 정책은 모델·시점에 따라 바뀔 수 있으므로 단정하지 않는다. 필요 시 공식 문서에서 최신 정책을 확인할 것 — 특히 대화 로그 보존 기간, Projects 업로드 문서의 보존/삭제 정책은 이 문서에 하드코딩하지 않는다.
