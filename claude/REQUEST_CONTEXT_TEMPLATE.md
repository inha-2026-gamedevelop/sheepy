# 작업 목적 설명 템플릿

> 목적: 작업 의뢰문에 "왜 하는지"(배경·기대 산출물·연관 시스템·과거 이력)를 담는 표준 컨텍스트 양식 제공 | 공식 근거: Anthropic 프롬프팅 가이드(의도 전달 패턴) + 프로젝트 내부 규칙 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

## 1. 이 문서의 역할 — PROMPT_TEMPLATE_STANDARD와의 구분

| 문서 | 담당 질문 | 내용물 |
|---|---|---|
| [프롬프트 표준 템플릿](PROMPT_TEMPLATE_STANDARD.md) | **무엇을** 하는가 | 작업 지시 본문 양식(요청·제약·완료 조건) |
| **이 문서** | **왜** 하는가 | 지시 앞에 붙는 컨텍스트 블록(배경·산출물·연관 시스템·이력) |

두 문서는 의뢰 양식 2부작이다. 완성된 의뢰문 = **[이 문서의 컨텍스트 블록] + [PROMPT_TEMPLATE_STANDARD의 지시 본문]** 순서로 조립한다.

**공식 근거**: 프롬프팅 가이드는 요청만 던지는 것보다 의도를 함께 전달하는 패턴 — "I'm working on [큰 목표] for [대상]. They need [산출물로 가능한 것]. With that in mind: [구체 요청]" — 을 권고하며, 모델이 의도(intent)를 이해하면 관련 정보를 연결하는 효율이 향상된다고 명시한다.

## 2. 언제 쓰고, 언제 생략하는가

과잉 지시 제거 원칙과 충돌하지 않도록, 이 템플릿은 **모든 요청에 강제하지 않는다**.

**써야 하는 경우** (이 저장소 기준):
- 신규 기능·시스템 작업 — 예: 보스 감정 전환 구동부, 씬 이관 로직처럼 여러 파일에 걸치는 작업
- `RewindManager`(리와인드 버퍼)·`GameDB`(SO 데이터)·`CameraManager` 등 **여러 시스템 경계를 넘는 작업** — 배경 없이는 모델이 영향 범위를 못 판단한다
- 과거 세션 결과물을 이어받는 작업 — `claude/HANDOVER.md`나 `claude/PLAN.md`에 이미 항목이 있는 주제

**생략해도 되는 경우**: 오타·단일 파일 조회·컨벤션 기계 적용(중괄호, 전위 증감 등) 같은 단발 요청. 이런 요청에 4개 필드를 채우는 것이 오히려 과잉 지시다.

## 4. 필드별 작성 요령 (이 저장소 실사례 기준)

**배경** — "어느 씬/페이즈에서 누가 쓰는가"가 빠지면 모델은 기능의 활성 조건을 추측한다. 보스 1페이즈 즉사 기믹 작업에서 "레이저가 나올 때 즘, 슬로우를 걸면 안전구역이 보인다"는 한 줄이 `Phase1State.CoTelegraphLaser`의 슬로우 연동 조건과 직결됐다.

**기대 산출물** — "완성"의 정의를 세션 범위로 쪼갠다. 보스 gap 로드맵 작업은 "P1(감정 전환+씬 이관)은 이번 세션, P2(스펙 해석 확인)는 사용자 답변 후, P3(폴리시)는 로드맵만"으로 명시적으로 분할했기 때문에 세션 종료 판정이 가능했다. 검증 방법(코드 리뷰만인지, 에디터 Play 라이브 검증까지인지)도 여기 적는다 — 검증 표준은 [테스트·검증 표준](TEST_AND_VERIFICATION_STANDARD.md) 참조.

**연관 시스템** — 위 체크리스트는 이 저장소에서 실제로 중요했던 지점만 모은 것이다:
- 리와인드: `claude/CLAUDE.md` Critical Rule 3·4 — 버퍼 용량은 `RewindManager.TickCapacity`만, 랜덤 패턴은 결정 로그. 감정 자동 전환 구현 시 `Phase1State.BuildSequence`/`Phase2State._waveXLog` 패턴을 그대로 따랐다.
- GameDB: 신규 필드는 `*DataSO.cs`에 추가하고 대응 `.asset`에도 기본값을 기입해야 인스펙터에서 즉시 보인다 — 이번 세션에 `GimmickCameraOrthoSize`, `EmotionInterval` 추가 시 실제로 둘 다 처리했다.
- 수정 범위: `claude/CLAUDE.md` Critical Rule 2 — 팀원 코드는 네임스페이스 없음, 수정 전 소유자 확인.

**과거 이력** — 메모리 파일은 `C:\Users\inha\.claude\projects\d--unity2d\memory\`에, 저장소 문서는 `claude/`에 있다. 파일명을 정확히 적으면 모델이 Read 한 번으로 이전 세션 결정 사항을 복원한다. 어떤 문서를 언제 읽는지는 [반복 참조 문서 인덱스](PROJECT_KNOWLEDGE_INDEX.md), 메모리 운영 규칙은 [메모리·교훈 규칙](MEMORY_RULES.md) 참조.

## 5. 실전 예시 — 보스 감정 자동 전환 의뢰문 비포/애프터

### 비포 (전형적인 요청문)

> 보스 감정 시스템 이어서 작업해줘. 자동으로 바뀌게 하면 돼.

이 요청으로 시작하면 모델이 스스로 재발견해야 하는 것들: `SetEmotion` 호출처가 Phase3의 Angry뿐이라는 것, 화남은 3페이즈 고정이라 자동 전환 후보에서 빼야 한다는 것, 리와인드 되감기 후에도 같은 감정 순서가 재현돼야 한다는 것(결정 로그 패턴 필요), 신규 GameDB 필드는 `.asset`에도 값을 넣어야 한다는 것.

### 애프터 (템플릿 적용)

> ## 작업 컨텍스트
>
> **배경**
> - 보스 감정(검정/하양/남색/핑크/파랑/화남)은 반사/낙뢰배율/혼란을 변조하지만, 현재 `SetEmotion`은 Phase3의 Angry/None만 호출해 대부분이 사문 코드다.
> - 기획: `claude/README.md` "보스 - Azathoth" 섹션, `claude/PLAN.md` Boss 섹션 HIGH 갭.
>
> **기대 산출물**
> - `BossController`에 주기적 랜덤 전환 코루틴 추가. 완료 판정: 전환 로그(또는 HUD)로 확인 + 되감기 후 순서 재현.
> - 이번 세션 범위: 전환 구동부 구현. 후속: 씬 배선(HUD 아이콘 등록)은 별도.
>
> **연관 시스템**
> - [x] 리와인드: 감정도 되감기 재현 필요 → 결정 로그(`Phase1State.BuildSequence` 패턴 참고) 필수
> - [x] GameDB: 전환 간격 필드 신설 → `BossDataSO.cs` + `BossDB.asset` 양쪽에 반영
> - [ ] 카메라/사운드: 이번 범위엔 해당 없음
> - [x] 수정 범위: `Minsung.Boss` 네임스페이스, 팀원 코드 접촉 없음
>
> **과거 이력**
> - 저장소 문서: `claude/PLAN.md` Boss 섹션 218-219행(감정 전환 구동부 갭 서술)
> - 미커밋: 없음(세션 시작 시점)
>
> 위 컨텍스트 하에: 감정 자동 전환 구동부를 구현해줘. (이하 [프롬프트 표준 템플릿](PROMPT_TEMPLATE_STANDARD.md) 지시 본문)

### 차이가 만드는 것

| 항목 | 비포 | 애프터 |
|---|---|---|
| 화남 제외 여부 | 모델 추측 → 재작업 위험 | 의뢰문에서 확정 |
| 리와인드 재현 | 작업 중 발견 → 재설계 | 착수 전 결정 로그 패턴 지정 |
| GameDB 짝 필드 | 필드만 추가하고 에셋 누락 위험 | 양쪽 반영 명시 |
