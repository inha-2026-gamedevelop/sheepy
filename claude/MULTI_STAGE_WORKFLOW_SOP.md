# 다단계 작업 SOP

> 목적: 여러 세션·여러 커밋에 걸치는 대형 작업을 Phase로 분할하고, Phase별 완료 정의(DoD)·롤백·문서화 의무를 표준화한다. | 공식 근거: Anthropic 프롬프팅 가이드(진행 보고의 툴 결과 근거화, fresh-context 검수 subagent) + 프로젝트 내부 규칙 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

이 문서는 이 저장소에서 실제로 진행한 대형 작업(보스 기획 gap 로드맵)에서 귀납한 절차다. 새 대형 작업을 시작할 때 이 문서를 먼저 읽고, 아래 틀에 작업을 끼워 넣은 뒤 착수한다.

---

## 0. 적용 대상 — 언제 이 SOP를 켜는가

다음 중 하나라도 해당하면 "다단계 작업"으로 선언하고 이 SOP를 적용한다.

| 판정 기준 | 이 저장소의 실례 |
|---|---|
| 3개 이상 시스템/페이즈를 관통하는 변경 | 보스 감정 전환+씬 이관(BossController/Phase2State/Phase3State/BossHandoff 동시 변경) |
| 리와인드 버퍼 구조에 지장 | 감정 결정 로그 도입 시 `BossFrame` 스냅샷 확장 |
| 씬 전환/빌드 설정 변경 | 2->3페이즈 씬 로드(`SceneManager.LoadScene`) 도입 |
| 한 세션 안에 끝나지 않을 것이 확실 | 보스 기획 gap 전체(감정/씬이관/스펙정합/폴리시 4단계로 로드맵화) |
| 되돌리기 어려운 조작 포함 | 3페이즈 전용 맵 씬으로 기존 `Boss.unity` 대체(명진 담당, 아직 미착수) |

단일 버그 수정·단일 파일 편집은 이 SOP 대상이 아니다 — 그런 작업까지 Phase화하는 것은 최소 변경 우선 원칙 위반이다(예: 슬로우모션 히트스톱 버그 수정은 이 SOP 없이 단일 커밋으로 처리했다).

## 1. 실사례 — 보스 기획 gap 로드맵 (P1 → P2 → P3)

근거: `claude/PLAN.md` Boss 섹션, 이번 세션 대화 이력.

| Phase | 내용 | 상태 | 왜 여기서 끊었나 |
|---|---|---|---|
| P1 | 스펙 핵심 미구현 — 감정 자동 전환 구동부, 2->3페이즈 씬 이관 | 구현 완료, `feature/boss-emotion-scene-handoff` 커밋 | 사용자 확인이 필요한 결정(전환 방식, 씬 이관 방식)이 명확히 갈리는 지점이라 구체 설계 전 AskUserQuestion으로 확정 |
| P2 | 스펙 해석/정합 확인 — 2페이즈 근접 주체, 1페이즈 기믹 트리거, `_totalHealth` 값 | 미착수(사용자 확인 대기) | 기획 해석이 갈리는 항목이라 코드보다 질문이 우선 |
| P3 | 폴리시/연출 — 낙뢰 리와인드 스냅샷, 애니메이터 완전 연동, 4페이즈 전용 패턴 | 로드맵만(리소스·기획 확정 후) | 팀원(진욱=애니메이터, 명진=맵) 리소스 의존이라 이 세션에서 착수 불가 |

이 사례가 남긴 SOP 핵심 규칙:

- **결정이 갈리는 지점은 Phase 착수 전에 질문한다.** P1 착수 전 "감정 전환을 주기적 랜덤으로 할지 페이즈 기반으로 할지", "씬 전환을 실제 씬 로드로 할지 인게임 맵 교체로 할지"를 AskUserQuestion으로 먼저 확정했다. 질문 없이 진행했으면 되돌릴 가능성이 컸다.
- **팀원 리소스 의존 항목은 별도 Phase로 격리한다.** P3의 애니메이터 연동·맵 제작은 이 세션이 통제할 수 없는 외부 의존이므로, 코드로 해결 가능한 P1과 섞지 않고 로드맵(HANDOVER "해야 할 것")으로만 남겼다.
- **Unity MCP 미연결 상태에서는 "검증"과 "코드 리뷰"를 분리 표기한다.** P1 완료 보고에서 "코드는 준비됨, Play 검증은 사용자 확인 필요"를 매번 명시했다.

## 2. 일반화 — Phase 분할 기준

이 사례에서 귀납한 분할 축. 새 작업의 Phase 경계는 아래 중 하나 이상과 일치해야 한다.

1. **결정 경계**: 사용자 확인이 필요한 갈림길(구현 방식 A/B)은 그 자체로 Phase 경계다. 확인 없이 다음 Phase로 넘어가지 않는다.
2. **위험 격리 경계**: 되돌리기 어려운 조작(기존 씬 대체, 리와인드 버퍼 구조 변경)은 독립 Phase로 뽑아 **가장 뒤에** 배치하거나, 선행 조건(팀원 산출물)이 갖춰진 뒤로 미룬다.
3. **검증 수단 경계**: 코드 리뷰로 확인 가능한 부분과 에디터 Play가 필요한 부분을 다른 Phase로 나눈다(Unity MCP 미연결 시 특히 중요).
4. **소유권 경계**: 팀원(명진/진욱) 담당 산출물에 의존하는 작업은 별도 Phase로 분리해 HANDOVER에 "해야 할 것"으로 담당자와 함께 남긴다. [ACTION_BOUNDARIES.md](ACTION_BOUNDARIES.md) 참조.
5. **데이터/코드 경계**: GameDB 필드 추가(데이터)와 그 필드를 소비하는 로직 변경(코드)은 가능하면 별도 커밋으로 나눈다 — 실제로 이번 세션에서 `chore: Common Boss GameDB 밸런싱 필드 확장`을 `feat: Boss ...` 커밋들과 분리했다.

Phase 계획을 세울 때 단계별 마이크로 지시를 나열하지 말 것 — 각 Phase는 "목표 + 제약 + DoD"만 정의하고 실행 방법은 열어 둔다.

## 3. Phase별 완료 정의 (DoD)

Phase 하나가 "완료"이려면 아래 3게이트를 전부 통과해야 한다. 진행 보고는 각 게이트의 **툴 결과 또는 사용자 확인을 근거로만** 작성한다([PROGRESS_CLAIM_POLICY.md](PROGRESS_CLAIM_POLICY.md)).

### 게이트 1 — 컴파일
- Unity MCP 연결 시 `read_console`로 에러 0 확인. 미연결 시 코드 리뷰로 문법 확인 후 "확인 필요" 명시. "코드를 저장했다"는 컴파일 확인이 아니다.

### 게이트 2 — 기능 검증
- 대상 씬에서 Play 검증. 표준 절차는 [TEST_AND_VERIFICATION_STANDARD.md](TEST_AND_VERIFICATION_STANDARD.md).
- 긴 다단계 작업의 Phase 종료 검수는 자기비판 대신 **fresh-context 검수 subagent**를 고려할 수 있다(사용자 요청 시) — 검수자 프롬프트는 [VERIFIER_SUBAGENT_SPEC.md](VERIFIER_SUBAGENT_SPEC.md).

### 게이트 3 — 문서화
- Phase 결과(커밋 해시, 잔여 작업, 발견 사항)를 `claude/HANDOVER.md`에 기록하기 전에는 Phase 완료를 선언하지 않는다.

## 4. 롤백 계획 — Phase 착수 전에 정한다

| 롤백 수단 | 이 프로젝트에서의 적용 |
|---|---|
| **Phase 커밋 분리** | 감정 전환/씬 이관/GameDB 필드 확장을 별도 커밋으로 나눠, 문제 발생 시 해당 커밋만 revert 가능 |
| **브랜치 격리** | git-flow `feature/<기능명>` 브랜치에서 작업해 `develop`/`main`을 오염시키지 않음 |
| **되돌리기 어려운 조작의 사전 조건** | 기존 `Boss.unity`를 새 보스맵으로 대체하는 작업은 새 맵이 검증된 뒤에만 진행 — `_phase3SceneName`을 임시값(`"Boss"`)으로 유지해 언제든 폴백 가능하게 설계 |
| **리와인드 정합 보호** | 결정 로그 패턴(`_emotionLog`+커서)은 기존 `Phase1State`/`Phase2State` 검증된 패턴을 그대로 재사용해 신규 버그 표면을 최소화 |

롤백 계획이 "git revert 하면 됨" 한 줄로 끝나지 않는 작업(씬 자산 교체, 리와인드 버퍼 구조 변경)은 Phase 계획서에 롤백 절차를 명문화한 뒤 착수한다.

## 5. 문서화 의무 — 무엇을, 언제

| 시점 | 의무 | 대상 |
|---|---|---|
| Phase 계획 확정 시 | Phase 표(목표/DoD/롤백/검증 수단) 작성 | 대화 내 계획 또는 `claude/PLAN.md` |
| **Phase 완료 직후** | 커밋 해시·검증 결과·발견 사항·잔여 작업 갱신 | `claude/HANDOVER.md`("이번에 고친 것"), 필요 시 메모리(규칙: [MEMORY_RULES.md](MEMORY_RULES.md)) |
| **세션이 작업 도중 끝날 때** | HANDOVER 갱신 — "다음 세션에서 이 문서를 읽고 이어서 작업할 것" | `claude/HANDOVER.md`(시점·표준 목차: [CHECKPOINT_RULES.md](CHECKPOINT_RULES.md)) |
| 담당 경계 밖 작업이 남을 때 | 담당자 이름과 함께 "해야 할 것"에 명시 | `claude/HANDOVER.md` (실례: "(명진) 맵 3종 제작", "(진욱) 보스 애니메이터 연동") |
| `claude/CLAUDE.md`에 반영할 사실이 생길 때 | 사용자 승인 하에 직접 반영 (이 프로젝트는 원본 직접 수정을 허용하는 완화된 방침 — [MEMORY_RULES.md](MEMORY_RULES.md) 9절 참고) | `claude/CLAUDE.md` |
