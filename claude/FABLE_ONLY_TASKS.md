# 최상위 모델 전용 작업 목록

> 목적: 이 프로젝트의 최고난도 작업을 최상위 모델에만 배정하고, 나머지는 중간/하위 모델로 내리는 배분 기준을 정의한다. | 공식 근거: Anthropic 모델 발표·가격 문서 + 프롬프팅 가이드 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식 (Fable 5 전용 → 모델 비종속 기준으로 일반화)

---

## 1. 왜 분류가 필요한가

모델별 정확한 가격·컨텍스트 수치는 세대가 바뀔 때마다 갱신되므로 이 문서에 하드코딩하지 않는다 — 필요 시 공식 가격 페이지나 `claude-api` 스킬로 조회한다. 일반 원칙은 유지된다:

- 상위 모델일수록 단가가 높다. "상위 모델이 아니면 실패하거나, 실패 시 복구가 비싼 작업"에만 쓴다.
- 공식 가이드는 "낮은 effort도 이전 세대 모델의 최고 effort 능력을 초과한다"고 명시한다 — 즉 이 프로젝트 백로그의 **대다수 작업은 중간/하위 모델로 충분**하다는 것이 공식 전제다.
- 모델×effort 조합의 세부 기준은 [Effort 사용 기준표](EFFORT_POLICY.md), 대형 파일 취급은 [컨텍스트 한도 기준](CONTEXT_LIMITS_POLICY.md) 참조.

## 2. 판단 기준 4가지

작업을 최상위 모델에 올릴지 판단할 때 아래 4가지를 순서대로 묻는다. **3개 이상 "높음"일 때만 최상위 모델 배정.**

| # | 기준 | 질문 | 이 프로젝트의 "높음" 사례 |
|---|---|---|---|
| 1 | **복잡도** | 다중 시스템이 교차하고, 전체를 동시에 머리에 올려야 하는가? | 보스 감정 자동 전환 + 씬 이관 — `BossController`/`RewindManager`/`Phase2State`/`BossHandoff`/`CameraManager`가 동시에 얽힘 |
| 2 | **가치** | 산출물이 일회성 수정이 아니라 이후 세션·팀이 재사용할 골격인가? | `claude/commit-convention.md`, `.claude/skills/clarify-ambiguous-request/`처럼 팀 전체가 계속 쓰는 규범 문서 |
| 3 | **실행가능성** | 하위 모델이 시도하면 부분 실패→재시도 루프로 오히려 비싸지는가? | 104개 스크립트 전수 스캔 기반 감사, 여러 하위 문서(30종)를 일관된 논리로 재작성하는 작업 |
| 4 | **오류비용** | 틀렸을 때 되돌리기 비싼가? | 리와인드 버퍼 구조 변경 — 참여자 간 인덱스가 어긋나면 전 시스템이 조용히 깨진다. 씬 YAML 직접 편집 — fileID 참조 오류는 에디터에서만 드러남 |

`claude/CLAUDE.md`의 Critical Rules(리와인드 버퍼, 네임스페이스 소유권)가 이미 기준 4를 규범 수준에서 강제하고 있다 — 이 문서는 그 위에 "모델 선택" 축을 추가하는 것이다.

## 3. 분류표

### 3-1. 최상위 모델 고난도 작업 (이 프로젝트 실사례 기준)

| 작업 | 핵심 근거 파일 | 기준 충족 |
|---|---|---|
| **보스 시스템 크로스컷 설계** | `BossController.cs`, `Phase1~4State.cs`, `RewindManager.cs`, `BossHandoff.cs` | 복잡도·가치·오류비용 높음 — 여러 페이즈 상태와 리와인드 훅이 동시에 얽힘 |
| **버그 원인이 증상과 분리된 진단** | 예: 슬로우모션 히트스톱 코루틴 인터럽트 — 증상(Shift 무반응)과 원인(다른 파일의 재시작 로직)이 멀리 떨어짐 | 복잡도·오류비용 높음 |
| **다수 문서의 일관된 재작성** | `claude/`의 AI 운영 문서 30종처럼 상호 참조가 많은 문서군을 일관되게 재작성 | 복잡도·가치·실행가능성 높음 |
| **기획-코드 전수 gap 감사** | 보스 기획서 vs `03.Boss/` 구현 전수 대조, 결과를 우선순위 로드맵으로 변환 | 복잡도·가치 높음 |
| **리와인드/GameDB 구조 변경** | `RewindManager.TickCapacity`, `IRewindable` 참여 규칙 변경 | 오류비용 매우 높음(참여자 전원에 영향) |

### 3-2. 하위 모델로 충분한 작업

| 등급 | 작업 | 근거 |
|---|---|---|
| **중간 모델** | 이미 승인된 설계를 다중 파일에 반복 적용하는 작업(예: GameDB 필드 추가 + 대응 에셋 값 기입), PLAN 문서 갱신, 단일 버그 수정 | 설계는 끝났고 선례 패턴을 따라가는 작업 — 복잡도 중간, 오류비용 중간 |
| **경량 모델** | 커밋 실행(`claude/commit-convention.md` 형식이 문서화됨), 단일 씬/프리팹 YAML 수정, GameDB 값 튜닝 | 절차가 문서화돼 있어 따라 하면 되는 작업 |
| **최경량 모델** | 코딩 컨벤션 기계적 대조(중괄호·전위 연산자 등 — `claude/coding-convention.md` 표 기준), 특정 심볼·경로 검색, 커밋 메시지 초안 | 단일 파일·단일 규칙 대조. 좁은 컨텍스트로 충분 |

**분류 원칙**: "코드를 새로 설계"하면 위로, "문서화된 절차를 실행"하면 아래로. 경계가 애매하면 한 단계 위 모델이 기본값이다(최상위 모델이 아니라).

## 4. 운용 수칙

- **effort**: 3-1급 작업은 `xhigh`(코딩·에이전트 권장값). 3-2급 작업에 상위 모델을 쓰게 되더라도 `medium` 이하로.
- **의뢰 형식**: 단계별 지시를 늘어놓지 말고 목표·제약·배경(왜 하는지)만 전달 — 과잉 처방은 오히려 성능을 깎는다. 양식은 [프롬프트 표준 템플릿](PROMPT_TEMPLATE_STANDARD.md)·[작업 목적 설명 템플릿](REQUEST_CONTEXT_TEMPLATE.md).
- **검증**: 장시간 자율 실행은 자기비판 대신 fresh-context 검증 서브에이전트를 고려할 수 있다(사용자 요청 시 — [검수 Subagent 사양서](VERIFIER_SUBAGENT_SPEC.md)). "완료" 선언은 툴 결과 또는 사용자 확인 근거 의무 — [진행상황 검증 규칙](PROGRESS_CLAIM_POLICY.md).
- **경계**: 팀원 소유 코드와 리와인드 버퍼 구조 변경은 착수 전 명시 확인 — [작업 경계선](ACTION_BOUNDARIES.md).
