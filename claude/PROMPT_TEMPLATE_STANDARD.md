# 프롬프트 표준 템플릿

> 목적: 작업 의뢰 프롬프트를 5필드(목적·범위·제약·완료 기준·검증 방법) 표준으로 통일해, 어떤 모델이 받아도 같은 품질의 작업이 나오게 한다. | 공식 근거: Anthropic 프롬프팅 가이드(platform.claude.com) + 프로젝트 내부 규칙 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

---

## 0. 핵심 원칙 — 왜 "단계 지시"가 아니라 "5필드"인가

공식 프롬프팅 가이드의 직접 반영 사항 3가지:

1. **단계 처방 금지, 목표·제약 기술**: "1) 파일 열고 2) 함수 고치고 3) 테스트해" 식의 step-by-step 나열은 오히려 성능을 떨어뜨릴 수 있다(과잉 지시 제거 권고). 경로 결정은 모델에게 맡기고, 의뢰자는 **도착점(완료 기준)과 울타리(범위·제약)** 만 정의한다.
2. **의도(왜)는 별도 양식에 위임**: 배경·의도를 함께 주면 성능이 오르지만("I'm working on X for Y…" 패턴), 본 템플릿을 비대하게 만들지 않기 위해 배경 서술은 [작업 목적 설명 템플릿](REQUEST_CONTEXT_TEMPLATE.md) 1~3문장을 프롬프트 앞머리에 붙이는 방식으로 분리한다.
3. **CoT 강요 문구 금지**: "추론 과정을 먼저 보여줘" 류 문구는 불필요한 제약이며 도움이 되지 않는다.

추가 수칙: `claude/CLAUDE.md`와 영속 메모리(중괄호 필수, `_camelCase`, GameDB 규칙 등)는 세션 시작 시 이미 로드된다. **프롬프트에 이 규칙들을 재복사하지 말 것** — 프롬프트에는 그 작업에만 고유한 정보만 쓴다.

## 1. 공통 5필드 정의

| 필드 | 기입 내용 | 이 프로젝트의 기준 |
|---|---|---|
| **목적** | 완료 후 상태 1~2문장 (절차가 아니라 결과) | 배경·의도는 [REQUEST_CONTEXT_TEMPLATE.md](REQUEST_CONTEXT_TEMPLATE.md)로 |
| **범위** | 폴더(`01.Scripts/`의 번호 폴더) / 네임스페이스 / 씬 | 기본 경계: **`Minsung.*` 네임스페이스는 민성 전용, 팀원(명진/진욱) 코드는 소유자 확인 후** → [ACTION_BOUNDARIES.md](ACTION_BOUNDARIES.md) |
| **제약** | 리와인드 버퍼 영향 여부, GameDB 필드 추가 여부, 팀원 코드 접촉 여부 등 | 리와인드 버퍼는 `RewindManager.TickCapacity`만 사용(직접 계산 금지) |
| **완료 기준** | 관찰 가능한 결과의 체크리스트 | "완료" 선언 전 증거 규칙: [PROGRESS_CLAIM_POLICY.md](PROGRESS_CLAIM_POLICY.md) |
| **검증 방법** | 대상 씬에서의 재현 경로 | [TEST_AND_VERIFICATION_STANDARD.md](TEST_AND_VERIFICATION_STANDARD.md) — Unity MCP 연결 여부에 따라 경로가 갈린다 |

### 필드 작성 수칙

- **모르는 필드는 비워두되 `미정 — 판단 후 질문` 으로 표기**한다. 억지로 채운 잘못된 제약이 빈 칸보다 해롭다.
- **리와인드 시스템에 영향이 "있음" 또는 "불확실"이면 반드시 "착수 전 질문" 문구를 쓴다.** `claude/CLAUDE.md` Critical Rule 3·4가 1차 방어선이지만, 프롬프트에 명시하는 것이 안전하다.
- **팀원(명진/진욱) 소유 코드 수정이 필요한 작업은 제약 필드에 "소유자 확인" 여부를 명기**한다. 무기입 시 기본값은 확인 전 보류다.

## 2. 템플릿 A — 버그 수정

### 빈 템플릿 (복붙용)

```
[버그 수정 의뢰]
배경: (REQUEST_CONTEXT_TEMPLATE.md 양식 1~3문장)
목적:
증상: 기대 동작= / 실제 동작=
재현 경로: (씬, 조작 순서)
범위: 씬= / 네임스페이스= / 수정 가능 폴더=
제약: 리와인드 영향=(없음 | 있음·불확실→착수 전 질문) / 팀원 코드 수정=(금지 | 소유자 확인 후) / 기타=
완료 기준:
검증 방법:
```

### 기입 예시 — 슬로우모션 히트스톱 버그 (2026-07-11 실사례)

```
[버그 수정 의뢰]
배경: 보스 1페이즈처럼 히트스톱(HitStopController)이 연쇄로 재요청되는 근접전 구간에서
      슬로우모션(Shift)이 전혀 반응하지 않는다는 재현 보고를 받았다.
목적: 히트스톱이 연쇄 발생해도 슬로우모션이 정상 반영되게 한다.
증상: 기대 동작=Shift로 즉시 Time.timeScale이 슬로우 배율로 전환 /
      실제 동작=1페이즈 근접전 중에는 아무 반응 없음
재현 경로: Boss.unity Play → 1페이즈 분신 근접전 중 Shift 홀드
범위: 씬=Boss.unity / 네임스페이스=Minsung.TimeSystem /
      수정 가능=Assets/01.Scripts/04.TimeSystem/HitStopController.cs
제약: 리와인드 영향=없음(리와인드 버퍼와 무관한 코루틴 로직) /
      팀원 코드 수정=불필요(민성 소유 파일) / 기타=Unity MCP 미연결, 컴파일 확인은 사용자 요청
완료 기준: (1) 근접전 중에도 Shift로 슬로우모션 반영 (2) 히트스톱 자체 기능 회귀 없음
검증 방법: 위 재현 경로대로 Play → 사용자가 직접 확인
```

> 실제 결말: 원인은 `HitStopController.Play()`가 코루틴을 정지 후 재시작(`UtilCoroutine.CheckRunCoroutine`)해, 연쇄 재요청 시 이전 코루틴이 복원 코드에 도달하지 못하고 `Time.timeScale`이 눌러앉는 것이었다. 종료 목표 시각만 뒤로 미루는 단일 코루틴 방식으로 수정 완료.

## 4. 템플릿 C — 분석·조사

경계 명시 원칙: 조사 의뢰에서 **"수정 금지, 평가만 보고하고 정지"를 명시하지 않으면** 모델이 선의로 수정까지 진행할 수 있다. 분석 템플릿에서 이 줄은 생략 불가다. 팀원 코드는 **읽기는 허용, 수정만 소유자 확인 필요**이므로 조사 범위에는 팀원 코드를 포함할 수 있다.

### 빈 템플릿 (복붙용)

```
[분석·조사 의뢰]
배경: (REQUEST_CONTEXT_TEMPLATE.md 양식 1~3문장)
목적: (답을 원하는 질문 1~3개)
조사 범위: (읽기 대상 폴더·키워드 — 팀원 코드 포함 가능)
제약: 코드·에셋 수정 일절 금지 — 원인/옵션 보고 후 정지. 수정은 별도 의뢰로.
산출물: (대화 보고 | claude/PLAN.md 갱신 | 메모리 기록)
완료 기준:
검증 방법: 모든 주장에 [파일:라인](경로#L라인) 근거 첨부
```

### 기입 예시 — 보스 기획-코드 gap 감사 (실사례)

```
[분석·조사 의뢰]
배경: 사용자가 보스 전체 기획서(공통 규칙+감정 6종+낙뢰+1~4페이즈)를 제시했다.
      현재 구현 상태와 얼마나 갭이 있는지 파악이 필요하다.
목적: (1) 기획서 대비 무엇이 구현/미구현/미완인가 (2) 우선순위 높은 갭은 무엇인가
조사 범위: Assets/01.Scripts/03.Boss/ 전체 (Phase1~4State, BossController, BossEmotion,
      각종 패턴 클래스) — 읽기 전용
제약: 코드 수정 일절 금지 — 원인과 갭 목록만 보고하고 정지.
산출물: 대화 보고 + claude/PLAN.md Boss 섹션 갱신
완료 기준: 기획 항목별 구현 상태(구현됨/미구현/미완) 표, 우선순위(P1/P2/P3) 로드맵
검증 방법: 모든 판정에 파일:라인 근거 첨부
```

> 실제 결말: 감정 전환 구동부 부재, 2->3페이즈 씬 전환 미구현이 P1으로 식별됐고, 사용자 확인 후 별도 의뢰(템플릿 B)로 구현했다.

## 5. 운영 규칙

- **모델·effort 선택은 프롬프트가 아니라 정책 문서로**: 작업 유형별 권장은 [EFFORT_POLICY.md](EFFORT_POLICY.md), 최상위 모델이 필요한 작업 판별은 [FABLE_ONLY_TASKS.md](FABLE_ONLY_TASKS.md).
- **다단계 작업**: 한 프롬프트에 Phase 여러 개를 욱여넣지 말고 [MULTI_STAGE_WORKFLOW_SOP.md](MULTI_STAGE_WORKFLOW_SOP.md)의 Phase 분할 규칙을 따른다. Phase당 템플릿 1장이 원칙이다.
- **답변 형식**: 결과 보고의 형식(결론 우선, 파일:라인 링크)은 [FINAL_RESPONSE_STYLE_GUIDE.md](FINAL_RESPONSE_STYLE_GUIDE.md)가 규정하므로 프롬프트에서 반복 지시하지 않는다.
- 상시 유효한 금지 항목(팀원 코드 소유권 등)은 [ACTION_BOUNDARIES.md](ACTION_BOUNDARIES.md)에 총정리돼 있다. 제약 필드에는 그 작업에서 실제로 건드릴 위험이 있는 항목만 골라 쓴다.
