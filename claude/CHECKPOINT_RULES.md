# 체크포인트 규칙

> 목적: 장기 작업이 세션 경계를 넘을 때 `claude/PLAN.md`를 언제·어떤 목차로 갱신하는지 성문화한다. | 공식 근거: Anthropic 프롬프팅 가이드(파일 기반 메모리 표면 제공 시 성능 향상, 진행 보고는 툴 결과 근거 의무화) + 프로젝트 내부 규칙 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

---

## 1. 왜 체크포인트인가

이 프로젝트의 장기 작업(보스 시스템 gap 로드맵, GameDB 마이그레이션 등)은 한 세션에 끝나지 않는다. 세션이 끊기면 대화 로그의 맥락은 소실되고, 다음 세션이 이어받을 수 있는 것은 **저장소에 남은 파일뿐**이다. 그 파일이 `claude/PLAN.md`다.

공식 가이드는 "파일 기반 메모리 표면을 제공하면 성능이 향상"되고, 메모리는 "한 줄 요약 + 이유가 포함된 markdown 파일"로 저장하라고 권고한다. 판단 기준은 단 하나다:

> **"다음 세션이 이 대화 로그 없이, 저장소 파일만 읽고 작업을 재개할 수 있는가?"**
> — 불가능하면 지금이 PLAN 갱신 시점이다.

## 2. 이 프로젝트의 PLAN 관례 — 단일 파일

**중요한 프로젝트 고유 차이**: 이 저장소는 `claude/PLAN_<주제>.md` 식 다중 파일이 아니라 **`claude/PLAN.md` 단일 파일**을 계속 갱신하는 방식을 쓴다(`claude/CLAUDE.md` Documentation Map에 그렇게 등록돼 있다). 새 작업을 시작했다고 새 PLAN 파일을 만들지 않는다 — 기존 파일을 그 시점 기준으로 다시 쓴다.

`PLAN.md`는 누적 아카이브가 아니라 현재 재개 지점이다. 완료된 작업의 상세 이력은 커밋 기록으로 남기고, 문서에는 진행 중인 항목과 다음 작업만 유지한다.

## 3. PLAN 갱신 트리거 (언제 쓰는가)

| 트리거 | 설명 |
|---|---|
| **T1. 대형 작업 중간 중단** | Phase/단계 경계에서 멈출 때. 사용자 컨펌·기획 확인 대기 포함 |
| **T2. 세션 한계 임박** | 컨텍스트 사용량이 커져 다음 세션으로 넘겨야 할 때 (분리 시점 판단은 [대화 분리 기준](CONVERSATION_SPLIT_RULES.md)) |
| **T3. 담당자 교대** | 팀원(명진/진욱) 소유 영역 작업을 넘기거나, 다음 세션에 이어받을 사람이 바뀔 때 |
| **T4. 작업 보류/원복** | 문제로 되돌렸지만 나중에 재작업할 때 — 실패 이력이 곧 자산 |

작성하지 않아도 되는 경우: 한 세션에 완결되고 커밋 메시지로 충분한 작업(예: 단순 버그 수정 1건). 이 경우 PLAN 갱신 없이 커밋 메시지만으로 충분하다.

`claude/CLAUDE.md` Critical Rule 11도 이미 "커밋 전 `claude/PLAN.md`를 최신 상태로 갱신"을 명시하고 있다 — 즉 **커밋 시점은 항상 PLAN 갱신 트리거**다.

## 4. 표준 목차 템플릿

```markdown
# 인수인계 문서 (YYYY-MM-DD)

> 브랜치: `<현재 작업 브랜치>` (git-flow 사용 시)

## 이번에 고친 것
1. **<기능/수정 요약>** (`관련 클래스/파일`) — 무엇을 왜 어떻게 고쳤는지 1~3문장. 검증 상태 포함(완료/미검증)

## 해야 할 것
- [ ] **<담당자 있으면 (이름)>** <작업 항목> — 왜 필요한지, 어디부터 시작하는지
```

- **"이번에 고친 것"**: 요구 증거 등급([PROGRESS_CLAIM_POLICY.md](PROGRESS_CLAIM_POLICY.md))을 함께 적는다. "Play 로그로 확인됨" vs "미검증"을 구분.
- **"해야 할 것"**: 담당자가 정해져 있으면 `(명진)`/`(진욱)`처럼 이름을 붙인다(이번 세션 실제 사례 — 맵 제작은 명진, 애니메이터 연동은 진욱으로 명시).
- 세부 이력(왜 이 구조를 선택했는지, 실패했던 접근 등)이 많이 필요하면 `claude/PLAN.md`의 해당 섹션(HIGH/MED/LOW)에 위임하고 PLAN.md에는 포인터만 남긴다.

## 5. 메모리(project_*)와 PLAN의 역할 분담

두 계층은 쌍으로 운용한다.

- **PLAN만 있고 메모리가 없으면** → 새 세션이 자동 로드하는 것은 메모리 인덱스(`C:\Users\inha\.claude\projects\d--unity2d\memory\MEMORY.md`)뿐이므로, "이 저장소는 `claude/PLAN.md`부터 읽는다"는 사실 자체가 메모리에 없으면 다음 세션이 놓칠 수 있다 — 다만 이 사실은 `claude/CLAUDE.md` Documentation Map에 이미 있으므로 대체로 문제없다.
- **메모리만 있고 PLAN이 없으면** → 요약 몇 줄로는 재개 절차·삽질 회피가 불가능하다.

| 정보 | 메모리 `project_*.md` (요약·포인터) | PLAN (상세 절차) |
|---|---|---|
| 현재 상태 한 줄 | O | 머리 블록쿼트/"이번에 고친 것"에 동일 상태 |
| 안전장치·경계 | O — 예: "팀원 소유 코드는 소유자 확인 후" | `claude/CLAUDE.md` Critical Rules에 이미 상시 명시 |
| 핵심 문서 포인터 | O — **필수.** 예: "보스 작업 재개 시 claude/PLAN.md부터" | — |
| 절차·표·스키마·코드 스니펫 | X (두지 않는다) | O |

**동기화 규칙**: PLAN의 상태를 갱신하면 같은 턴에 관련 메모리 항목의 상태 문구도 갱신한다. 메모리 작성 형식 자체의 규칙은 [메모리·교훈 규칙](MEMORY_RULES.md)을 따른다.
