# 문서 분석 워크플로우

> 목적: 기획 문서/구두 스펙을 코드와 대조해 갭 분석 → 구현 계획으로 변환하는 표준 절차 | 공식 근거: 프로젝트 내부 규칙(실사례: 보스 기획서 대조 분석) | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

---

## 0. 적용 범위와 기준 실사례

기획서(문서/구두 설명)를 받아 "무엇을 만들어야 하고, 현재 무엇이 있고, 무엇이 비어 있는가"를 산출하는 모든 작업에 적용한다. 이 워크플로우의 기준 실사례는 **보스(Azathoth) 기획 대조 분석**이다:

- 입력: 사용자가 대화로 제시한 보스 전체 기획서(공통 규칙 + 감정 6종 + 낙뢰 + 1~4페이즈 상세) — 저장소 어디에도 없던 구두/채팅 스펙.
- 출력: `claude/PLAN.md` Boss 섹션(구현 완료/HIGH/MED/LOW 갭)과 대화 내 요약 표.
- 대상 코드: `Assets/01.Scripts/03.Boss/` 전체(`BossController`, `Phase1~4State`, `BossEmotion`, 각종 패턴 클래스).

절차는 5단계다: **① 기획 원문 구조화 → ② 현행 코드 갭 분석 → ③ 리와인드 영향 판정 → ④ 미결 질문 목록 → ⑤ 구현 계획**. 단계 산출물과 진행 상태는 `claude/PLAN.md`에 누적한다(작성 시점·목차 기준은 [체크포인트 규칙](CHECKPOINT_RULES.md) 참조).

---

## 1단계 — 기획 원문 구조화

기획서를 코드 단위와 1:1로 대응 가능한 **시스템/페이즈 단위**로 분해한다. 서술 순서가 아니라 구현 단위로 재편성하는 것이 핵심.

- 보스 사례: "공통 규칙(체력/공격/타이머) / 감정 6종 / 낙뢰 / 1페이즈 / 2페이즈 / 3페이즈 / 4페이즈"로 재편성 — 흩어진 서술(예: 감정별 효과가 여러 문단에 나뉨)을 담당 클래스(`BossEmotion`/`BossEmotionExtensions`) 기준으로 묶었다.
- 각 단위에 **담당 클래스 후보**를 병기한다(예: "1페이즈 즉사 기믹 — `Phase1State`"). 후보가 없으면 그 자체가 갭이다.
- 원문이 대화 로그에만 있다면 이 단계에서 수치·규칙을 문서로 옮겨 적는다(보스 사례: 페이즈 경계 64000/48000/32000/16000/0, 감정별 배율 등을 `claude/PLAN.md`에 옮김) — 이후 세션이 원문 대화 없이 작업을 이어갈 수 있게 하기 위함.

## 2단계 — 현행 코드 갭 분석

기획 도메인별로 **어느 코드와 대조할지**가 정해져 있다. 이 프로젝트의 결정표:

| 기획 도메인 | 대조 코드/클래스 | 대조 GameDB(SO) |
|---|---|---|
| 보스 체력/페이즈 | `BossController.cs`(TakeDamage, PhaseFloorHealth) | `BossDataSO.TotalHealth/PhaseCount` |
| 감정 | `BossEmotion.cs`, `BossEmotionExtensions.ShouldReflect/LightningRateMultiplier` | `BossDataSO`(ConfusionInterval 등) |
| 낙뢰(공통 패턴) | `BossLightningPattern.cs`(IBossPattern 구현) | `BossDataSO.Lightning*` |
| 페이즈별 기믹 | `Phase1State.cs`~`Phase4State.cs` | `BossDataSO.Gimmick*`, `Phase2Wave*`, `Phase3Laser*` |
| 리와인드 참여 | 각 `*State`의 `RecordTick/ApplyRewindTick/OnRewindEnd` | 해당 없음(구조적 확인) |

대조 시 확인할 것:

1. **로직이 있는지, 도달 가능한지 구분**: 감정 반사/낙뢰배율 로직은 `BossEmotionExtensions`에 이미 구현돼 있었지만, `SetEmotion` 호출처가 Phase3의 Angry뿐이라 **다른 감정은 코드는 있어도 도달 불가능한 사문 코드**였다. "구현됨"과 "실제로 발동함"을 갭 분석표에서 분리 판정한다 — 이 구분이 보스 gap 분석에서 가장 중요한 발견이었다.
2. **GameDB 필드 존재 여부 확인**: `BossDataSO`에 이미 필드가 있는지, 없다면 신규 추가가 필요한지(예: `EmotionInterval`, `GimmickCameraOrthoSize`는 이번 세션에 신규 추가).
3. **갭 태그 4종 분류**(보스 사례에서 확립, 표준으로 채택):
   - `[코드]` C# 로직만으로 해결 (에디터 없이도 진행 가능)
   - `[씬배선]` 씬/프리팹에 오브젝트 배치·와이어링 필요 (`_phase1Clones`/`_body`/`_heartPickup` 등)
   - `[에셋]` 리소스(맵, 애니메이션) 필요 — 팀원 산출물 대기 가능성
   - `[기획확인]` 사양 모호 — 4단계 미결 질문으로 승격
   이 태그는 "Unity 에디터 접근 없이" 진행 가능한 작업(`[코드]`)을 골라내는 필터로도 쓰였다.

## 3단계 — 리와인드 영향 판정 (사전 질문 규칙 발동점)

기획 요구가 "되감기 후에도 재현되어야 하는 상태"를 포함하면 리와인드 참여 구조 변경이 필요한지 판정한다. **영향이 있으면 구현 전에 반드시 사용자에게 질문한다**(또는 최소한 명시적으로 결정 로그 설계를 제시한다).

판정 기준:

- 기획 요구를 만족하는 랜덤/상태 변화가 현행 `BossFrame`(또는 각 State의 리와인드 스냅샷)에 없고, 결정 로그 없이 되감기 후 다른 결과가 나올 수 있으면 → **리와인드 구조 확장 대상**.
- 보스 감정 사례: 감정 자동 전환은 랜덤이므로, 결정 로그(`_emotionLog`+커서) 없이 구현하면 되감기 후 다른 감정 순서가 나와 리와인드 일관성이 깨진다 — `BossFrame`에 감정+커서를 추가해 해결했다.
- 리와인드 버퍼는 `RewindManager.TickCapacity`만 사용해야 한다(`claude/CLAUDE.md` Critical Rule 3) — 직접 계산 금지.
- 세이브 시스템은 이 프로젝트에 **해당 없음**이므로, 다른 프로젝트의 "세이브 스키마 영향 판정"에 대응하는 것이 이 프로젝트에서는 "리와인드 스냅샷 영향 판정"이다.

## 4단계 — 미결 질문 목록

`[기획확인]` 태그가 붙은 항목을 질문 형태로 모아 사용자에게 회신 요청한다. 단정하거나 임의 구현하지 않는다. 보스 사례의 실제 미결 질문:

1. "2페이즈 근접 주체는 기획상 '1페이즈 분신 패턴 유지'인데 코드는 본체(Body)가 수행 — 어느 쪽이 맞는가?"
2. "1페이즈 기믹 트리거는 기획 '분신 2체 모두 처치' vs 코드 '총 피통 하한 도달' — 오버킬 시 정합 확인 필요."

질문에는 "현행이 이렇다"는 근거(클래스/코드 상태)를 붙여, 사용자가 코드 현황을 모른 채 답하지 않게 한다. 질문 회신 전까지 해당 항목은 5단계 계획에서 **보류(P2로 고정)** 처리한다(실제로 이번 세션에서 이 두 항목을 P2로 분류했다).

## 5단계 — 구현 계획

갭 분석표를 우선순위 목록으로 변환한다. 보스 사례의 정렬 기준을 표준으로 채택한다:

1. **스펙 핵심 미구현(P1)**: 기획의 핵심 기능인데 코드가 아예 없거나 도달 불가능한 것 — 감정 전환 구동부, 씬 이관.
2. **스펙 해석/정합(P2)**: 기획 문구와 코드가 다르게 해석될 수 있는 것 — 사용자 확인 후 착수.
3. **폴리시/연출(P3)**: 리소스·기획 확정이 선행돼야 하는 것 — 로드맵만 남기고 착수하지 않는다.

계획이 여러 세션에 걸치면 Phase 분할·DoD 정의는 [다단계 작업 SOP](MULTI_STAGE_WORKFLOW_SOP.md), 각 Phase 완료 검증은 [테스트·검증 표준](TEST_AND_VERIFICATION_STANDARD.md)을 따른다. 후속 작업을 의뢰서로 넘길 때는 [프롬프트 표준 템플릿](PROMPT_TEMPLATE_STANDARD.md)과 [작업 목적 설명 템플릿](REQUEST_CONTEXT_TEMPLATE.md)을 사용한다.

## 산출물 템플릿 — 갭 분석표

`claude/PLAN.md`의 해당 시스템 섹션마다 아래 형식을 채운다. "현행 상태"는 반드시 Read/Grep으로 확인한 사실만 적고, 추정이면 `(추정)`을 명기한다.

```markdown
## N. <시스템/페이즈명> — 담당: `<클래스명>` (없으면 "담당 클래스 없음")

| 기획 요구 | 현행 상태 | 갭 | 조치 |
|---|---|---|---|
| <기획 문장/수치 그대로> | <클래스 확인 결과. "로직 O / 발동 X" 등 층위 구분> | <없음 / 부분 / 전체> | `[코드]`/`[씬배선]`/`[에셋]`/`[기획확인]` + 한 줄 방안 |

**리와인드 영향**: 없음 | 있음(대상: ___, 결정 로그/스냅샷 확장 필요)
**미결 질문**: (없으면 "없음")
```

작성 예시 (보스 감정 실사례에서 발췌·재구성):

```markdown
## 감정 자동 전환 — 담당: `BossController`

| 기획 요구 | 현행 상태 | 갭 | 조치 |
|---|---|---|---|
| 검정/하양/남색/핑크/파랑이 주기적으로 바뀜 | 반사/낙뢰배율 로직 O(BossEmotionExtensions) / 자동 전환 발동 X | 전체 | `[코드]` CoEmotionLoop 신설 + 결정 로그 |
| 화남은 3페이즈 고정 | Phase3State.Enter에서 SetEmotion(Angry) 호출 O | 없음 | 자동 전환 후보에서 제외만 확인 |

**리와인드 영향**: 있음 (감정 전환도 되감기 재현 필요 → BossFrame에 감정+커서 스냅샷 추가)
**미결 질문**: 없음(주기적 랜덤 방식으로 사용자 확정)
```
