# 진행상황 검증 규칙

> 목적: AI가 "완료"·"수정됨"을 선언하기 전에 반드시 확보해야 하는 증거의 종류와 등급을 정의한다. | 공식 근거: Anthropic 프롬프팅 가이드 "진행 보고 근거화"(툴 결과 기반 claim 감사 지시 시 fabricated status report 급감) | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

---

## 1. 원칙: "완료"는 증거가 있는 상태 서술이다

공식 프롬프팅 가이드는 장시간 자율 실행 중 진행 보고 전 **각 주장(claim)을 실제 도구 결과(tool result)와 대조해 감사(audit)** 하도록 지시하면 조작된 상태 보고가 사실상 0%로 떨어진다고 명시한다. 이 규칙을 본 프로젝트의 도구 체계(Unity MCP 연결 시/미연결 시, 사용자 Play 확인)에 맞게 구체화한 것이 이 문서다.

이 프로젝트에서 실제로 이 원칙이 필요했던 사례:

- **슬로우모션 히트스톱 버그**: `HitStopController`의 코루틴 재시작 로직은 **정적 분석만으로는 확정하기 어려운** 문제였다 — 사용자가 "Phase1에서 안 먹는다"고 재현 결과를 알려준 뒤에야 원인(연쇄 재요청 시 복원 코드 미도달)을 특정할 수 있었다. "코드가 논리적으로 맞아 보인다 = 버그 없음"이 성립하지 않는 사례.
- **Unity MCP 미연결**: 이번 세션은 Unity 에디터에 직접 접근할 도구가 없다. 이 상태에서 "Play모드에서 확인했습니다"라고 쓰는 것은 근거 없는 주장이다 — 실제로는 매번 "확인 부탁드립니다"로 명시했다.

## 2. 필수 증거 3종 — "완료" 선언 전 확보 순서

### 2-1. 컴파일 무오류 (모든 코드 수정의 최소선)

- **Unity MCP 연결 시**: 스크립트 수정 후 `mcp__unity__refresh_unity`(도구명은 실제 연결된 MCP 서버 기준)로 에셋 리프레시 → 컴파일 완료 대기 → `read_console`을 Error 필터로 호출해 **에러 0건**을 도구 출력으로 확인.
- **Unity MCP 미연결 시(이번 세션 기본 상태)**: 코드 리뷰로 문법·타입·네임스페이스 오류 여부를 스스로 재확인하고, **"컴파일 확인은 에디터에서 직접 부탁드립니다"** 라고 명시한다. "컴파일됩니다"를 도구 출력 없이 단정하지 않는다.

### 2-2. 기능/플로우 확인 (로직 변경 시)

- 기준: 관련 씬(예: `Boss.unity`)에서 Play 후 실제 동작을 확인.
- 에이전트는 Play 화면을 직접 볼 수 없다(MCP 미연결 시 특히). 이 경우 **"미검증 — 에디터 Play모드 확인 필요"로 명시**하고 완료라 말하지 않는다. 실제로 이번 세션에서 감정 전환/슬로우모션/씬 이관 모두 이 방식으로 사용자에게 확인을 요청했다.
- MCP 연결 시 절차 상세는 [테스트·검증 표준](TEST_AND_VERIFICATION_STANDARD.md) 참조.

### 2-3. 데이터 무결성 확인 (GameDB/씬 자산 변경 시)

로컬 세이브 시스템은 이 프로젝트에 **해당 없음**(Supabase 백엔드로 랭킹/고스트 리플레이만 처리, `Assets/StreamingAssets/KEY.txt` 필요 — 민감정보 취급은 [민감정보 기준](DATA_RETENTION_AND_PRIVACY_RULES.md) 참조). 대신 이 프로젝트에서 무결성이 중요한 것은:

1. **GameDB(SO) 필드-에셋 동기화**: `BossDataSO`(또는 다른 `*DataSO`)에 필드를 추가하면 대응 `.asset` YAML에도 반드시 기본값을 기입한다. 필드만 추가하고 에셋을 안 건드리면 Unity 인스펙터에서 기본값(C# 초기화식)으로 채워지긴 하지만, 팀원이 즉시 인식 못 할 수 있으니 diff로 짝을 확인한다.
2. **리와인드 버퍼 인덱스 정합**: `RewindManager.TickCapacity` 외의 방식으로 버퍼를 만들지 않았는지 확인(CLAUDE.md Critical Rule 3).
3. **씬 YAML 참조 무결성**: 씬 파일을 직접 편집했다면 fileID/GUID 참조가 깨지지 않았는지, 에디터에서 Missing 컴포넌트로 나타나지 않는지 확인 요청.

## 3. 증거 등급표

| 등급 | 이름 | 정의 | 인정 근거 |
|---|---|---|---|
| 0 | 주장만 | 도구 출력 없이 서술 | **없음 — 진행 보고에 사용 금지** |
| 1 | 컴파일 통과 | 에러 0건 확인 | MCP `read_console`(Error 필터) 결과, 또는 미연결 시 코드 리뷰 + "확인 필요" 명시 |
| 2 | 기능 확인 | 해당 기능 1회 동작 관찰 | Play 중 로그/사용자 확인 보고 |
| 3 | 플로우 검증 | 정규 플로우(씬 진입부터) 통과 | MCP 로그 또는 사용자의 명시적 확인 답변 |

### 작업 유형별 요구 등급

| 작업 유형 | 이 프로젝트의 예시 | 요구 등급 |
|---|---|---|
| MD 문서·주석·리포트만 | `claude/` 문서 작성 | 등급 외 — 단, 인용 경로 전부 Glob/Read 실존 확인 |
| 로그 문구·수치 튜닝·주석 정리 | `Debug.Log` 문구, GameDB 기본값 | 1 |
| 코드 로직 수정 | 코루틴/상태머신/패턴 로직 | 2 이상 |
| 씬 YAML 직접 편집 | `Boss.unity` 오브젝트 배치 | 2 + 에디터 재임포트 확인 요청 |
| 페이즈 전환·씬 로드 경로 | `BossController.Start`/`BossHandoff` 복원 로직 | 3 |
| 리와인드 스냅샷/결정 로그 | `RecordTick`/`ApplyRewindTick`/`OnRewindEnd` | 3 (되감기 왕복 확인 필요) |

등급 3이 필요한데 2까지만 확보된 상태로 턴을 마쳐야 한다면, 보고에 **"등급 2까지 검증됨, 플로우 검증 미실시"** 를 명시한다. 검증을 대신할 fresh-context 검수자 활용은 [검수 Subagent 사양서](VERIFIER_SUBAGENT_SPEC.md) 참조.

## 4. 시스템 프롬프트 삽입용 표준 문구

공식 권고("검증되지 않으면 명시 + 실제 도구 출력으로 실패/스킵 보고 + 완료·검증 시 hedging 없이 명시")를 이 프로젝트 용어로 번안한 문구. [SYSTEM_PROMPT_V2.md](SYSTEM_PROMPT_V2.md) 또는 장기 자율 작업 의뢰서에 그대로 삽입한다.

```text
[진행 보고 규칙 — 근거 없는 진행 보고 금지]
- 진행상황이나 완료를 보고하기 전, 각 주장을 실제 도구 결과(tool result)와 대조해 감사한다.
- 검증되지 않은 항목은 "미검증"이라고 명시한다. 검증된 것처럼 서술하지 않는다.
- 실패했거나 건너뛴 단계는 실제 도구 출력을 근거로 그대로 보고한다. 성공으로 포장하지 않는다.
- 완료되었고 검증까지 마친 항목은 얼버무림 없이 명확히 "완료(검증됨)"라고 말한다.
- 이 프로젝트의 최소 증거:
  · 컴파일 = Unity MCP read_console 에러 0건(연결 시) / 미연결 시 코드 리뷰 + "확인 필요" 명시
  · 기능   = Play모드 관찰 로그(MCP) 또는 사용자의 명시적 확인 답변
  · 데이터 = GameDB 필드-에셋 짝 확인, 리와인드 버퍼는 TickCapacity 정합 확인
- 수정한 모든 위치를 [파일:라인](경로#L라인) 하이퍼링크로 빠짐없이 제시한다.
```

## 5. 보고 형식 — 수정 파일 하이퍼링크 의무

코드를 수정했다면 **요청받지 않아도** 모든 수정 위치를 하이퍼링크로 제시한다.

- 형식: `[파일명:시작-끝라인](워크스페이스 상대경로#L시작-L끝)` — 예: `[HitStopController.cs:94-104](Assets/01.Scripts/04.TimeSystem/HitStopController.cs#L94-L104)`
- **수정한 메서드/블록 단위로 전부** 나열한다. 요약 한두 개만 링크하고 끝내지 않는다.
- 라인 번호는 **편집 후 최종 위치 기준**. 대량 편집 뒤에는 Grep으로 최종 라인을 재확인하고 링크한다. 추정 금지.
- 보고 직전 자가 점검: "이 턴에 Edit/Write한 위치 중 링크를 안 단 곳이 있는가?" + "링크한 각 항목의 증거 등급은 무엇인가?"
- 보고문 전체 스타일(결론 우선·한국어)은 [최종 답변 스타일 가이드](FINAL_RESPONSE_STYLE_GUIDE.md) 참조.

## 6. 상태 어휘 통일

진행 보고·PLAN 모두 동일 어휘를 쓴다.

| 표기 | 의미 |
|---|---|
| 완료(검증됨) | 요구 등급 충족 + 증거 확보 |
| 수정 완료(미커밋) | 코드 검증됨, git 커밋 전 |
| 구현됨(미검증) | 코드 작성됨, 등급 2 미달 — "완료"로 승격 금지 |
| 진단 완료·미수정 | 원인 규명만 끝남 |
| 승인·미구현 | 계획만 확정 (사용자가 계획을 승인했으나 아직 구현 전) |
| 검증 대기 | 구현 완료, 사용자 Play 확인 등 외부 조건 대기 |

미완 상태로 세션을 넘길 때의 기록 시점·목차는 [체크포인트 규칙](CHECKPOINT_RULES.md)을 따른다.
