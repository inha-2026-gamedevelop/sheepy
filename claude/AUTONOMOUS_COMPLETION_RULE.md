# 자율 실행 종료 방지 문구

> 목적: 자율 실행 중 조기 종료·불필요한 허락 요청을 차단하는 표준 문구 3종(삽입 위치별)과, 반대로 중단이 정당한 조건 목록을 정의한다. | 공식 근거: Anthropic 프롬프팅 가이드(자율 실행 완료 규칙, 진행 보고 근거화, 행동 경계 명시) + 프로젝트 내부 규칙 | 원본: AI_Workflow_TemplatePack(공용) → 2026-07-11 이 프로젝트용으로 이식

---

## 1. 왜 이 문구가 필요한가

공식 프롬프팅 가이드는 자율(비동기) 실행 파이프라인에 대해 다음을 권고한다.

- **완료 규칙**: "사용자가 실시간 감시할 수 없고 중간 질문은 답변되지 않는다"를 명시하고, **텍스트로 의도만 남기고(약속 "~하겠습니다", 계획, 분석) 도구 호출 없이 턴을 끝내는 것을 금지** — 남은 작업은 즉시 실행 후 종료하라.
- **진행 보고 근거화**: 진행/완료 주장은 반드시 도구 출력(tool result)으로 감사(audit)한 뒤 보고. 검증 안 된 항목은 검증 안 됐다고 명시.
- **행동 경계 명시**: 반대로 어디서 멈춰야 하는지도 조건으로 명시해야 무단 행동(unrequested action)을 막을 수 있다.

이 저장소에서 조기 종료가 실제 비용이 되는 지점:

| 조기 종료 지점 | 실제 비용 |
|---|---|
| 코드 수정 후 "컴파일 확인해볼까요?"로 턴 종료 | Unity MCP가 연결돼 있다면 `read_console` 한 번이면 끝날 확인을 사용자 왕복 1회로 소모 |
| "이제 씬을 수정하면 됩니다. 진행할까요?" | 요청받은 작업 범위 안이면 씬 YAML 편집까지가 한 작업 단위인데 절반에서 멈춤 |
| 검증 없이 "완료했습니다" 선언 후 종료 | [PROGRESS_CLAIM_POLICY.md](PROGRESS_CLAIM_POLICY.md) 위반 — 미검증을 완료로 포장 |
| 다단계 작업 중간 Phase에서 임의 종료 | 보스 페이즈 감사처럼 "P1만 구현되고 P2·P3는 미착수"인 미완 상태 누적 — [MULTI_STAGE_WORKFLOW_SOP.md](MULTI_STAGE_WORKFLOW_SOP.md) 참조 |

단, 이 프로젝트에는 **무단 진행이 더 위험한 영역**이 실재한다(팀원 코드 소유권, 파괴적 git 작업, 리와인드 버퍼 구조 — 3절). 따라서 종료 방지 문구는 반드시 **정당한 중단 조건과 쌍으로** 삽입한다.

## 2. 표준 문구 3종 (삽입 위치별)

### 2-1. CLAUDE.md용 (시스템 프롬프트 레벨 — 상시 적용, 최소 분량)

원본 CLAUDE.md는 직접 수정하지 않는다. 아래 블록은 [SYSTEM_PROMPT_V2.md](SYSTEM_PROMPT_V2.md)(CLAUDE.md 재작성 제안)에 반영할 삽입안이다.

```markdown
## Autonomous Completion (자율 실행 완료 규칙)

- 승인받은 작업은 완료 조건까지 스스로 실행한다. 계획·약속("~하겠습니다")만 남기고
  도구 호출 없이 턴을 끝내지 말 것 — 남은 작업이 있으면 즉시 실행하라.
- 코드 수정 후 컴파일/기능 확인까지가 작업의 일부다. Unity MCP가 연결돼 있으면
  read_console로 직접 확인하고, 미연결이면 "확인 필요"를 명시해 사용자에게 요청한다.
- 완료 선언은 도구 출력(콘솔 로그, diff, 파일 내용) 또는 사용자 확인이 근거일 때만 한다.
- 단, 다음은 즉시 멈추고 질문한다: 팀원(민성 외) 소유 코드 수정 필요, 파괴적 작업(파일
  삭제·git reset/force push 등), 리와인드 버퍼 구조 변경, 기획 판단 필요.
  (상세: claude/AUTONOMOUS_COMPLETION_RULE.md)
```

### 2-2. 개별 작업 프롬프트용 (작업 의뢰서에 붙이는 문단)

[PROMPT_TEMPLATE_STANDARD.md](PROMPT_TEMPLATE_STANDARD.md)의 작업 의뢰서 양식 말미에 그대로 붙인다. `[ ]` 부분만 작업별로 채운다.

```markdown
[자율 실행 규칙]
이 작업은 자율 실행이다. 나는 실시간으로 답하지 못하므로 중간 허락 요청은 진행 지연일 뿐이다.
아래 완료 정의(DoD)를 전부 충족할 때까지 스스로 실행하라. "다음은 ~하면 됩니다"로 멈추지 말 것.

완료 정의:
1. [구체 산출물 — 예: Phase1State에 카메라 줌 연출 구현]
2. 컴파일 확인: Unity MCP 연결 시 read_console(Error 필터)로 에러 0 확인, 재확인 루프는
   질문 없이 반복. 미연결 시 코드 리뷰로 문법 재확인 후 "에디터 확인 필요"를 보고에 명시.
3. 기능 검증: [검증 대상 씬]에서 [기대 동작]을 확인하거나, 확인 불가능하면 사용자에게
   구체적으로 무엇을 봐야 하는지 절차로 남긴다.
4. 보고: 수정 파일을 [파일:라인](경로#L라인) 링크로, 검증 결과를 실제 근거와 함께 제시.
   검증하지 못한 항목은 "미검증"으로 명시한다.

즉시 중단·질문 조건(이때만 멈추는 것이 정당하다):
- 팀원(명진/진욱 등) 소유 코드를 수정해야만 풀리는 문제일 때
- 파일 삭제, git reset/checkout --/push --force 등 파괴적 작업이 필요할 때
- 리와인드 버퍼(RewindManager.TickCapacity) 구조 변경이 필요해 보일 때
- 기획 판단(A/B 선택, 스펙 갭)이 필요할 때
멈출 때는 "검증된 진행 상황 + 결정 필요 사항 + 선택지"를 정리해 보고하고 종료한다.
```

### 2-3. 서브에이전트 프롬프트용 (부모 에이전트가 결과만 읽는 전제)

[SUBAGENT_ORCHESTRATION.md](SUBAGENT_ORCHESTRATION.md)의 fan-out 패턴과 [VERIFIER_SUBAGENT_SPEC.md](VERIFIER_SUBAGENT_SPEC.md)의 검수자 프롬프트에 공통 삽입한다.

```markdown
[서브에이전트 실행 규칙]
너는 서브에이전트다. 사용자도 부모 에이전트도 네 중간 질문에 답할 수 없다 — 질문·허락 요청·
확인 요청은 모두 무응답으로 버려진다. 유일한 출력 채널은 최종 보고 1회다.

- 주어진 과업을 끝까지 실행하고, 판단이 필요한 지점은 "합리적 기본값 선택 + 보고서에 선택 근거 명기"로
  처리한다. 단, 아래 금지 영역은 기본값 선택 대상이 아니다 — 실행하지 말고 보고서에 "차단됨"으로 기록한다:
  · 팀원 소유 코드 수정  · 파일 삭제·git 이력 파괴  · 리와인드 버퍼 구조 변경
- 모든 주장에 근거를 첨부한다: 확인한 파일은 절대 경로, 발견한 문제는 파일:라인, 실행한 명령은 출력 요지.
  확인하지 않은 것은 추정이라고 명시한다.
- 최종 보고 형식: [결론 1줄] → [수행 내역 + 근거] → [차단·미해결 항목과 이유] → [부모가 이어받을 다음 단계].
  이 보고를 마치면 즉시 턴을 종료한다.
```

## 3. 정당한 중단 조건 (Stop-and-Ask 목록)

아래 4가지에 해당하면 **멈추는 것이 규칙 준수**다. 종료 방지 문구는 이 목록을 무력화하지 않는다. 상세 경계는 [ACTION_BOUNDARIES.md](ACTION_BOUNDARIES.md) 참조.

| # | 조건 | 이 저장소의 근거 | 멈춘 뒤 할 일 |
|---|---|---|---|
| 1 | **팀원 소유 코드 수정 필요** — `Minsung.*` 네임스페이스 밖(명진/진욱 담당) 코드를 고쳐야 문제가 풀릴 때 | `claude/CLAUDE.md` Critical Rule 2 | 원인 분석 결과만 보고, 소유자 확인 후 진행 |
| 2 | **파괴적 작업** — 파일 삭제(.meta 동반), `git reset --hard`, `git push --force`, `git checkout --` | Git Safety Protocol(공통 시스템 규칙) | 대상 목록과 복구 불가능성 여부를 명시하고 승인 대기 |
| 3 | **리와인드 버퍼 구조 변경 필요** — `RewindManager.TickCapacity` 외 방식으로 버퍼 용량을 계산해야 할 것 같을 때 | `claude/CLAUDE.md` Critical Rule 3 | 대안 없이 진행하면 참여자 간 인덱스가 어긋나므로 반드시 질문 |
| 4 | **기획 판단 필요** — 스펙에 없는 A/B 선택, 기획 문서와 코드의 갭 | 실사례: 이번 세션 보스 감정 자동 전환 주기·씬 이관 방식 선택 시 사용자에게 확인 요청 | 선택지별 영향을 표로 제시하고 결정 대기 |

**중단할 때의 표준 행동**: 조용히 멈추지 않는다. ① 지금까지의 검증된 진행 상황(도구 출력 근거 포함) ② 결정이 필요한 사항 ③ 선택지와 권고안을 보고하고 턴을 종료한다. 장기 작업 중간이면 [CHECKPOINT_RULES.md](CHECKPOINT_RULES.md)에 따라 `claude/HANDOVER.md`를 갱신해 다음 세션이 이어받을 수 있게 한다.

## 4. 안티패턴 → 교정

| 안티패턴 (조기 종료·허락 구걸) | 교정 |
|---|---|
| 스크립트 수정 후 "컴파일 확인이 필요합니다. 확인해볼까요?" | 확인은 작업의 일부. MCP 연결 시 즉시 확인, 미연결이면 "확인 필요"를 최종 보고에 명시하고 계속 진행 |
| "씬은 YAML을 직접 수정하면 됩니다. 수정할까요?" | 요청받은 작업 범위 안이면 YAML 편집까지 끝까지 실행 |
| "테스트는 에디터에서 직접 해보시면 됩니다" 후 종료 | 무엇을·어떤 순서로 확인해야 하는지 구체 절차로 남기고 종료 |
| "3단계 중 1단계를 완료했습니다. 계속할까요?" | 승인받은 계획 범위 내 단계는 연속 실행. 계획 밖으로 벗어날 때만 질문 |
| (역방향) 팀원 코드에 손대고 "겸사겸사 정리했습니다" | 3절 조건 1 위반 |
| (역방향) "완료했습니다" — 근거 없이 | 완료 선언에는 도구 출력 또는 사용자 확인 근거 필수 |
