01. Story

되감을수록, 기억은 흐려진다

어느 날 눈을 뜬 봉제인형 Sheepy. 손에 쥔 것은 깨진 회중시계 하나뿐,
자신이 왜 여기 있는지도, 무엇을 잃었는지도 모른다.

Sheepy는 한 아이와 함께 놀던 봉제인형이었다. 어느 날 버려졌고, 오랜 시간이 지나 살아났다.
시간을 되감아 그 시절로 돌아가려 하지만, 되감을수록 기억은 오히려 희미해진다.

보스를 처치하고 같이 아이와 놀았던 변했던 곰돌이를, 그리고 추억을 구해보자.
사진 속엔 아이와 Sheepy 그리고 곰돌이가 함께한 모습이 담겨있다.
02. Core Concept

스토리와 시스템이 같은 질문을 한다

게임의 핵심 메커닉인 타임 리와인드는 연출이 아니라, "되감을수록 기억이 희미해진다"는 스토리를 플레이 경험으로 번역한 장치다.

  • 분신 없이는 깰 수 없는 보스 패턴, 혼자서는 할 수 없는 조작이 존재한다 - 스킬은 연출이 아니라 필수 생존 수단
  • 플레이어가 리와인드하면 보스 체력도 함께 되돌아간다 - 시간 조작이 세계 전체에 동일하게 적용되는 규칙
  • 전투 타이머는 리와인드와 무관하게 계속 흐른다 - "되돌릴 수 없는 것"이 존재
03. Core Systems

3가지 핵심 시스템

타임 리와인드 (R)

최근 N초(기본 10초)간의 행동을 역재생 연출로 되감은 뒤 시작점으로 복귀

분신 소환

되감기 종료 시 기록된 커맨드 클립을 정방향 재생하는 분신 등장

슬로우모션 (Shift)

시간 배율 0.4배. fixedDeltaTime도 동일 비율 보정해 슬로우 중에도 물리가 부드럽다

05. Time System - 기록

테이프에 매 순간을 녹화한다

리와인드의 핵심은 사실 "되감는 순간"이 아니라 평소에 무엇을 기록해두는가다. 매 물리 프레임(FixedUpdate)마다 지금 상태를 캡처해, 카세트테이프처럼 생긴 RingBuffer(순환 버퍼)에 한 칸씩 쌓아 나간다.

Move

위치 / 속도 / 접지 여부

Attack?

이 틱에 공격했는지 + 차지공격 여부

Interact?

이 틱에 상호작용(레버 등)했는지

Hearts

이 틱 종료 시점의 체력(하트 반칸 단위)

Anim

재생 중이던 애니메이션 상태 스냅샷

이 5가지를 묶은 것이 한 틱의 TickCommand다. RingBuffer는 용량을 넘으면 가장 오래된 칸에 덮어쓰기(테이프 맨 뒤로 되감아 다시 녹화하는 것과 같음)만 하므로, 언제나 "최근 N초"만 정확히 기억한다.

// 버퍼 용량 = 몇 초를 기억할지 ÷ 물리 프레임 간격 TickCapacity = RecordSeconds(기본 10초) / Time.fixedDeltaTime // 매 FixedUpdate마다 참여자 전원이 한 칸씩 기록 buffer.Push(new TickCommand(move, hasAttack, attack, hasInteract, interact, hearts, anim));
06. Time System - Command Pattern

왜 좌표가 아니라 '명령'을 저장할까

단순하게 만든다면 "그 순간의 좌표"만 저장해도 위치는 되감을 수 있다. 하지만 그러면 공격의 피해 판정, 상호작용, 애니메이션은 재현할 수 없다. 그래서 Sheepy는 한 틱에 있었던 행동을 스스로 재생 방법을 아는 명령 객체로 캡슐화해서 저장한다 - 커맨드 패턴이다.

커맨드 = 캡슐화된 행동

  • MoveCommand.Apply(actor) - 위치/속도/접지를 그대로 적용
  • AttackCommand.Execute/Undo(actor) - 공격을 정방향/역방향으로 재생
  • InteractCommand.Execute(target) - 레버 등 상호작용 재연

ICommandActor - 받는 쪽은 몰라도 된다

커맨드를 받는 쪽은 SetPose() / PlayAttack() 딱 2개만 구현하면 된다. PlayerController와 CloneController(분신)가 똑같이 이 인터페이스를 구현하기 때문에, 같은 명령 리스트를 두 곳에서 재사용할 수 있다 - 본체는 거꾸로 재생하고, 분신은 처음부터 재생한다.

Execute (정방향 재생)

실제 게임 진행 중 공격 - 피해 + 모션이 함께 발생한다

Undo (역방향 재생)

되감기 스크럽 중 - 모션만 거꾸로 보여주고, 피해는 다시 발생시키지 않는다

07. Time System - 재생

되감기가 끝나면, 그 테이프가 분신이 된다

1
R 입력
잠겨 있지 않고 기록이 있으면 StartRewind()
->
2
역순 스크럽
매 FixedUpdate마다 여러 틱씩 건너뛰며 최신 -> 과거로 ApplyRewindTick() 방송
->
3
공격 Undo
건너뛴 구간에 공격 기록이 있으면 AttackCommand.Undo() - 모션만 역재생
->
4
테이프 처음(0) 도달
FinishRewind() - 참여자 전원에게 OnRewindEnd() 통지
->
5
분신 소환
방금 되감은 버퍼를 그대로 ClonePool.Spawn(buffer)에 전달

분신(CloneController)은 넘겨받은 커맨드 리스트를 처음 틱부터 순서대로 Execute하며 재생한다. 본체가 "역재생"할 때 쓴 것과 완전히 같은 데이터·같은 커맨드 타입이고, 방향과 Execute/Undo 선택만 다르다 - 이게 커맨드 패턴이 주는 재사용성이다.

16. Development Process - AI 활용

AI를 어떻게 활용했는가

Claude Code를 프로그래머의 상시 협업 도구로 썼다. 핵심 원칙은 "AI가 매번 처음부터 다시 설명을 들어야 하는 상태"를 피하는 것 - 그래서 문서 관리 자체를 개발 프로세스의 일부로 다뤘다.

문서 관리

  • claude/CLAUDE.md를 팀 공용 진입점으로 - 코딩 컨벤션·아키텍처·Critical Rules가 세션 시작마다 자동 로드되고, 근거 문서(coding-convention.md/UML.md/gamedb.md/boss.md)로 연결
  • gamedb.md·boss.md 같은 시스템별 문서는 "AI가 이 문서만 읽고 바로 작업할 수 있을 정도"로 자기완결적으로 작성 - 구조/사용법/함정/확장 절차까지 포함
  • 세션이 끝나도 남아야 할 결정(교정된 관례, 진행 상황)은 코드 옆 주석이 아니라 영속 메모리에 별도 기록 - 다음 세션이 재설명 없이 그대로 이어받는다

실제 사용 용도

  • 전역 일관성 리팩터 - 기능 하나를 폐기하면(예: LP·스킬게이지 제거) 코드·씬·ScriptableObject·문서를 한 번에 추적해 정리
  • 문서-코드 동기화 유지 - 코드가 바뀌면 README·CLAUDE.md·gamedb.md·UML.md도 같이 갱신해 "문서가 코드보다 낡는 상태"를 방지
17. Development Process - AI 활용 사례

구조를 제안받고, 모호한 요청은 되묻게 한다

아키텍처 구조 제안

기능을 그대로 구현해달라고만 하지 않고, 구조 자체를 AI에게 제안받아 채택한 사례들이다.

  • 코디네이터 패턴 - 비대해지던 단일 PlayerController를 Input/Movement/Combat/Interaction/Rewind 컴포넌트로 쪼개고, PlayerController는 조율만 하도록 재설계 제안 -> 채택
  • 싱글톤 공통 베이스 - 매니저마다 제각각이던 Awake 싱글톤 구현을 PersistentSingleton<T> / SceneSingleton<T>로 통일하자는 제안 -> OnSingletonAwake() 오버라이드 규칙으로 정착
  • RewindLockHandle 패턴 - 전역 bool 토글 대신 AcquireRewindLock(owner)가 반환한 핸들을 소유자가 Dispose()하는 방식을 제안 -> 여러 주체가 동시에 잠가도 안전한 구조로 채택

모호한 요청 명확화 스킬

.claude/skills/clarify-ambiguous-request - 요청이 모호할 때 AI가 추측으로 채우지 않고 먼저 코드/문서를 조사한 뒤, 정말 사용자만 결정할 수 있는 지점만 골라 되묻도록 만든 커스텀 스킬.

18. Development Process - 문서 예시 ①

claude/CLAUDE.md - Critical Rules

1. 코딩 컨벤션 준수: coding-convention.md가 최우선 - 매직넘버 금지 -> 밸런싱 수치는 GameDB, 코드 계약값은 Constants.*.cs - 모든 제어문 중괄호 필수(Allman), 전위 증감(++i) 3. 리와인드 버퍼 용량은 RewindManager.TickCapacity만 사용 - 직접 RecordSeconds / fixedDeltaTime 계산 금지 6. 리와인드 잠금은 핸들로 관리 - AcquireRewindLock(owner)의 반환값을 보유, Dispose()로 해제 7. 싱글톤 공통 베이스 사용 - PersistentSingleton<T> / SceneSingleton<T> 상속 - Awake 직접 구현 금지, OnSingletonAwake() 오버라이드

만약 지키지 않은 팀원이 있다면 커밋 하기 전 코딩컨벤션이 틀렸다고 하고 고쳐준다고 물어본다.
